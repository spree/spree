# Replace Taxons with Categories & Add Collections

**Status:** Category read/write surface shipped (5.5–5.6): `Spree::Category < Spree::Taxon` scoped subclass, direct `store_id` ownership + backfill, full Admin Categories API (CRUD + `reposition` + nested products/custom_fields/translations), and counter caches (`products_count`). **Remaining for 6.0:** table rename, the `Collection` stack (model + join + rules + API), model-level removal of automatic/rules from Category, and dropping `Taxonomy`.
**Target:** Spree 6.0 (Category surface landed in 5.5–5.6; 6.0 renames the table, splits out `Collection`, and drops `Taxonomy`)
**Depends on:** API v3 conventions, ProductType plan (6.0-product-types.md), Rich Text Descriptions plan (6.0-rich-text-descriptions.md), Admin SPA plan (6.0-admin-spa.md — dashboard UI), Channels/Publications plan (6.0-channels-catalogs-b2b.md — the `CollectionRules::AvailableOn` rule's channel-aware implementation depends on its `available_on` → `ProductPublication#published_at` migration; see Migration Phase 5)
**Author:** Damian + Claude
**Last updated:** 2026-07-09

## Summary

Split the overloaded `Taxon` model into two distinct concepts that every modern commerce platform separates:

- **Category** — hierarchical product classification (Men > Clothing > T-Shirts). A product's primary organizational home. Manual membership only.
- **Collection** — flat, curated or rule-based product groupings ("Summer Sale", "New Arrivals", "Best Sellers"). Supports both manual curation and automatic rules.

Drop the `Taxonomy` model entirely — it's a grouping layer that adds indirection without value. Categories become a single tree per store (rooted, nested set). Collections are flat, store-scoped, and own the current automatic/rule-based functionality.

This also renames `Classification` to `ProductCategory` and `TaxonRule` to `CollectionRule`.

## Problem

1. **"Taxon" and "Taxonomy" are jargon.** No other commerce platform uses these terms. Every new developer has to learn what they mean. The industry-standard separation is Category (hierarchical) + Collection (flat/curated).

2. **Taxon is overloaded.** The same model serves as both hierarchical categories AND flat collections (via the "Collections" taxonomy with automatic rules). These have fundamentally different semantics:
   - Categories are **hierarchical**, **manual**, and define a product's **primary classification**
   - Collections are **flat**, often **automatic** (rule-based), and used for **merchandising** (seasonal campaigns, "on sale", "new arrivals")

3. **Taxonomy adds indirection without value.** `Taxonomy` is essentially a named container for a tree of Taxons, scoped to a Store. The root Taxon mirrors the Taxonomy name (kept in sync via callbacks). In practice, merchants have 2-3 taxonomies ("Categories", "Brands", "Collections"). This grouping is replaced by separating Collections out entirely; brands are out of scope (see Key Decisions → Brands).

4. **`Classification` tells you nothing.** The Product↔Taxon join model name gives no hint about what it joins.

## Current State

Much of the **Category** half has already shipped (5.5–5.6). What exists today:

| Model | API name | Prefix | Notes |
|---------|----------|--------|-------|
| `Spree::Taxonomy` | n/a (internal) | `txnmy_` | Container for taxon trees, scoped to store. Being dropped. |
| `Spree::Taxon` | `categories` | `ctg_` | Nested set tree on `spree_taxons`; still carries `automatic` + `TaxonRule` (Collection's future home) |
| `Spree::Category` | `categories` | `ctg_` | `class Category < Taxon` with `default_scope { manual }` + `requires_taxonomy? = false`. **Not STI** — there is no `type` column; it shares `spree_taxons` rows with `Taxon` and is a scoped view whose ownership is driven by `store_id` instead of `taxonomy_id`. |
| `Spree::Classification` | n/a | n/a | Product↔Taxon join (table: `spree_products_taxons`) |
| `Spree::TaxonRule` | n/a | `txrule_` | STI: AvailableOn, Sale, Tag |

**Already shipped:**
- **`store_id` on `spree_taxons`** (nullable) + backfill task `spree:taxons:backfill_store_id`; `Spree::Category` owns via `store_id` directly (no taxonomy required, through `SingleStoreResource`/`ensure_store`).
- **Counter caches** on `spree_taxons`: `children_count`, `classification_count` (direct), and `products_count` (descendant-inclusive, deduped — recomputed, not a Rails counter_cache).
- **Store API**: read-only `categories` controller + `CategorySerializer` (product exposes `taxons` under the `categories` key, expand-gated).
- **Admin API**: full CRUD `categories` controller + `reposition` + nested `products` / `custom_fields` / `translations`; `Admin::CategorySerializer < CategorySerializer`.
- **Promotion rules** already speak `category` on the wire (`Promotion::Rules::Taxon#api_type => 'category'`, permits `category_ids`); the model rename is still pending.

**No longer true:** default taxonomies ("Categories", "Brands", "Collections") and automatic taxons ("On Sale", "New Arrivals") are **no longer auto-created per store** — `Store#ensure_default_taxonomies_are_created` / `ensure_default_automatic_taxons` were deprecated (2026-03-10) and removed in 5.5 (PR #14020, 2026-05-11). New stores get no taxonomies; categories are created on demand. The `taxonomy_*_name` locale keys remain (only `brand_taxon` still reads `taxonomy_brands_name`).

**Known gap — "manual" sort in the v3 Store API.** "Manual" sort means ordering products by their hand-set position *within the grouping* (`Classification.position` → `ProductCategory.position`). Today it works only in the admin nested products controller and the legacy `Spree::Products::Find` finder — it is **not** implemented in the v3 search-provider path: `SearchProvider::Database#apply_sort` has no `'manual'` branch and never receives the category/collection context, and the Meilisearch provider indexes no position and can't sort by a per-grouping (many-valued) join position. So a category page's default "manual" sort does not currently reflect the merchant's hand-ordering on the v3 API. This affects **collections too**: their dedicated endpoint still lists through the provider (the collection is applied as a filter, not an AR pre-scope), so it can't bypass the provider to order by position. The fix is provider-level and shared by both — see Open Questions.

## Remaining for 6.0

The Category read/write surface is done. What's left:

1. **Table rename** — `spree_taxons` → `spree_categories`, `spree_products_taxons` → `spree_product_categories` (+ FK columns), plus the inheritance flip (Category becomes the base class, `Taxon` becomes a deprecation alias). `spree_taxon_rules` is dropped (not renamed) — `spree_collection_rules` is a new table.
2. **The full `Collection` stack** — `Spree::Collection`, `Spree::ProductCollection` join, `Spree::CollectionRule` (STI), and the Store + Admin API. Nothing exists yet. (Dashboard UI is tracked in `6.0-admin-spa.md`.)
3. **Model-level split** — move `automatic`, `rules_match_policy`, `sort_order`, and `taxon_rules` off Category onto Collection (today they still live on `Taxon`/`Category` and are only excluded at the API layer).
4. **Drop `Taxonomy`** — remove the model, table, and the `taxonomy_id` column. Also removes `Product#brand_taxon` / `#brand_name` (brands are out of scope for 6.0 — see Key Decisions → Brands).

## Key Decisions (do not deviate without discussion)

### Category

- **`Spree::Taxon` renamed to `Spree::Category`**, `spree_taxons` renamed to `spree_categories`
- **`Spree::Taxonomy` dropped.** Categories are a single nested set per store. The `taxonomy_id` column becomes `store_id` directly on the category.
- **`Spree::Classification` renamed to `Spree::ProductCategory`**, table `spree_products_taxons` renamed to `spree_product_categories`, columns `taxon_id` → `category_id`
- **Categories are manual only.** The `automatic` flag, `rules_match_policy`, `sort_order`, and `taxon_rules` association move to Collection.
- **Prefix stays `ctg_`** (already in use)
- **Nested set stays.** `awesome_nested_set` with `parent_id`, `lft`, `rgt`, `depth`, `children_count`. Proven, no reason to change.
- **`pretty_name` and hierarchical `permalink` stay.** These are valuable for breadcrumbs and SEO.
- **`hide_from_nav` stays** on Category — useful for categories that exist for organization but shouldn't appear in navigation.

### Counter caches

Two product-count concepts, named consistently across grouping models. `products_count` means "products in this grouping, including descendants" — for a flat Collection that's just its direct members, so the name means the same thing on both models.

| Model | Column | Meaning | Maintained by |
|-------|--------|---------|---------------|
| Category | `products_count` | descendant-inclusive, deduped (self + subtree) | recompute (`recalculate_products_count`) |
| Collection | `products_count` | direct (flat) | Rails `counter_cache` on `ProductCollection` |
| Product | `categories_count` | # categories the product is in | Rails `counter_cache` on `ProductCategory` |
| Product | `collections_count` | # collections the product is in | Rails `counter_cache` on `ProductCollection` |

- **Drop `classification_count`** on both `spree_categories` (only the legacy Rails admin reads it; the dashboard/API use `products_count`) and `spree_products` (renamed to `categories_count`).
- The category's `products_count` is **not** a Rails counter_cache — it's recomputed on classification change / node move, so `ProductCategory` carries no category-side count counter.

### Collection

- **New model `Spree::Collection`** — flat (no tree), store-scoped
- **New join table `Spree::ProductCollection`** (replaces the automatic taxon's use of Classification)
- **Owns automatic/rule-based membership** — the `automatic` flag, `rules_match_policy`, and rules association move here from Taxon
- **`Spree::TaxonRule` renamed to `Spree::CollectionRule`** — STI subtypes `Tag`, `Sale`, `AvailableOn`, all registered in `config.spree.collection_rules` (which **mirrors the existing `config.spree.taxon_rules`**). `AvailableOn` ships with an interim legacy-column implementation; its channel-aware behavior **depends on `6.0-channels-catalogs-b2b.md`** (see Design Details → Automatic collection regeneration; Migration Phase 5)
- **Prefix: `coll_`**
- **Both manual and automatic collections.** Manual collections have `ProductCollection` records managed by admins. Automatic collections have `CollectionRule` records that regenerate membership.
- **Sort order on Collection**, not Category. The `SORT_ORDERS` constant (`manual`, `best_selling`, `price asc`, etc.) moves to Collection.

### Taxonomy → dropped

- **No replacement model, and no `kind` column.** Taxonomy simply goes away. With brands out of scope (see below), there's no remaining need to distinguish "kinds" of trees — a store has one category tree plus flat collections.
- **Store scoping moves to Category directly.** Every category has `store_id` (required). Children auto-copy from parent on create. Simple `where(store_id:)` scoping, no joins needed.

### Brands → out of scope (no brand feature in 6.0)

**Decision:** Spree 6.0 ships **no brand-specific construct** — no `Spree::Brand` model, no brand attribute/column, no `:brands` tag context, no category `kind` column. The going-forward model is just **Category** (hierarchy) + **Collection** (curated/rule-based); a store that wants to group products by brand models it as a regular Category or Collection like any other grouping.

- The existing `Product#brand_taxon` / `#brand_name` and the "Brands" taxonomy convention are **removed along with `Taxonomy`**, with no replacement. The brand lookup was the only remaining `Taxonomy`-dependent runtime path, so removing it is what unblocks dropping `Taxonomy`.
- The `taxonomy_brands_name` locale key is removed.
- Rationale: none of the surveyed platforms model brand as a category, and a proper first-class brand is a larger feature than 6.0 warrants. Rather than ship a half-measure (brand-as-category or brand-as-tag), 6.0 ships nothing brand-specific — a dedicated `Spree::Brand` can be designed later (core or extension) without a rushed decision here.

## Design Details

### Category model

```ruby
class Spree::Category < Spree.base_class
  has_prefix_id :ctg

  include Spree::TranslatableResource
  include Spree::TranslatableResourceSlug
  include Spree::Metafields
  include Spree::Metadata
  include Spree::MemoizedData

  acts_as_nested_set dependent: :destroy, counter_cache: :children_count

  # Store ownership via the shared concern — provides belongs_to :store, presence,
  # and ensure_store (children inherit the parent's store). Replaces the old
  # hand-rolled copy_store_from_parent; Taxon#set_store is deprecated for 6.0.
  include Spree::SingleStoreResource

  # Products
  has_many :product_categories, class_name: 'Spree::ProductCategory', dependent: :destroy_async
  has_many :products, through: :product_categories

  # ProductType integration (from 6.0-product-types.md)
  has_many :product_type_categories, class_name: 'Spree::ProductTypeCategory', dependent: :destroy
  has_many :product_types, through: :product_type_categories

  # Promotions
  has_many :promotion_rule_categories, class_name: 'Spree::PromotionRuleCategory', dependent: :destroy
  has_many :promotion_rules, through: :promotion_rule_categories

  # Attachments
  has_one_attached :image
  has_one_attached :square_image

  # Validations
  validates :name, presence: true, uniqueness: { scope: %i[parent_id store_id], case_sensitive: false }
  validates :hide_from_nav, inclusion: { in: [true, false] }

  # Translations
  TRANSLATABLE_FIELDS = %i[name pretty_name description permalink].freeze
  translates(*TRANSLATABLE_FIELDS, column_fallback: !Spree.always_use_translations?)
  # Interim: description via ActionText (current behavior — same as Collection).
  # 6.0-rich-text-descriptions.md owns migrating BOTH Category and Collection off
  # ActionText to sanitized HTML text columns; this plan does not change description storage.
  translates :description, backend: :action_text

  # Scopes
  scope :for_store, ->(store) { where(store_id: store.id) }

  # No automatic/rules — that's Collection's job
end
```

> **Interim state:** today `Spree::Category < Spree::Taxon` (shares `spree_taxons`, no `type` column). The rename **flips** this — Category becomes the base class on `spree_categories` and `Spree::Taxon` becomes a deprecation alias (see Migration Phase 3). The design above is the post-rename target.

### Collection model

```ruby
class Spree::Collection < Spree.base_class
  has_prefix_id :coll

  include Spree::SingleStoreResource
  include Spree::TranslatableResource
  include Spree::TranslatableResourceSlug
  include Spree::Metafields
  include Spree::Metadata

  # FriendlyId with history, mirroring Category/Taxon. `use: :history` records superseded
  # permalinks in the shared `friendly_id_slugs` table (polymorphic, scoped by sluggable_type)
  # so old collection URLs keep resolving after a RENAME — a deliberate SEO requirement.
  # No new table (friendly_id_slugs already exists for Taxon). Within-store uniqueness is
  # enforced separately by `validates :permalink, uniqueness: { scope: :store_id }` + the DB
  # index: a collision ERRORS (exactly as Taxon does — friendly_id is not scoped and does not
  # auto-suffix here). Generation mirrors Taxon's dual set_permalink — a model-level
  # `before_validation` (so the column_fallback base column is populated for the default
  # locale, which the Translation `before_save` alone misses) plus a Translation-level
  # `before_save` — but flat: `generate_slug` is `name.to_url` with no parent prefix.
  extend FriendlyId
  friendly_id :permalink, slug_column: :permalink, use: :history
  before_validation :set_permalink, if: :name

  RULES_MATCH_POLICIES = %w[all any].freeze
  # Space format (same values as Spree::Taxon::SORT_ORDERS) — required by the sort
  # pipeline: FiltersAggregator#to_api_sort splits on a space ('price asc' -> 'price'),
  # and apply_sort maps to 'price'/'-price'/'best_selling'. Underscores would break default_sort.
  SORT_ORDERS = [
    'manual', 'best_selling',
    'price asc', 'price desc',
    'available_on desc', 'available_on asc',
    'name asc', 'name desc'
  ].freeze

  acts_as_list scope: :store_id

  # store ownership (belongs_to :store + presence) comes from SingleStoreResource

  # Products
  has_many :product_collections, class_name: 'Spree::ProductCollection', dependent: :destroy_async
  has_many :products, through: :product_collections

  # Automatic rules. The association is named `rules` (not `collection_rules`) so the
  # Admin API's write key equals the serializer's read key: ParamsNormalizer derives the
  # flat nested-attributes key from the association name (`rules` → `rules_attributes`),
  # so a `rules: [...]` payload routes correctly (read/write symmetry — see CLAUDE.md).
  has_many :rules, class_name: 'Spree::CollectionRule', dependent: :destroy
  accepts_nested_attributes_for :rules, allow_destroy: true,
    reject_if: proc { |attrs| attrs['value'].blank? }

  # Attachments
  has_one_attached :image
  has_one_attached :square_image

  # Validations (store presence comes from SingleStoreResource)
  validates :name, presence: true
  validates :permalink, uniqueness: { scope: :store_id, case_sensitive: false, allow_blank: true }
  validates :rules_match_policy, inclusion: { in: RULES_MATCH_POLICIES }, presence: true
  validates :sort_order, inclusion: { in: SORT_ORDERS }, presence: true

  # Translations
  TRANSLATABLE_FIELDS = %i[name description permalink].freeze
  translates(*TRANSLATABLE_FIELDS, column_fallback: !Spree.always_use_translations?)
  # Interim: mirror the current Category — description via ActionText. This overrides the
  # "text column per 6.0-rich-text-descriptions.md" target for the initial build (those
  # primitives aren't shipped yet); that plan migrates BOTH Category and Collection off
  # ActionText together.
  translates :description, backend: :action_text

  scope :manual, -> { where.not(automatic: true) }
  scope :automatic, -> { where(automatic: true) }

  # Regenerate product membership from rules (moved from Taxon)
  def regenerate_products
    return unless automatic?
    Spree::Collections::RegenerateProducts.call(collection: self)
  end

  def products_matching_rules(opts = {})
    return Spree::Product.none unless automatic? && rules.any?
    # Same logic as current Taxon#products_matching_rules
  end
end
```

### Automatic collection regeneration

Automatic membership is materialized into `ProductCollection` rows (as today's `Classification` is for automatic taxons). It regenerates on **events** — a `CollectionRule` change, a `rules_match_policy` change, or a product save (via `AutoMatchCollections`). Event-driven regeneration has **no clock**, so time-relative rules drift: a product that ages past an `AvailableOn` window stays in the collection until some unrelated write re-triggers evaluation.

A scheduled job closes that gap for **time-based** rules:

- **Registry** — `config.spree.time_based_collection_rules = [Spree::CollectionRules::AvailableOn]` in `engine.rb`, **net-new** (no `taxon_rules` equivalent — taxons ship no scheduled refresh, hence their drift). This is a separate axis from the `config.spree.collection_rules` rule registry (which mirrors `config.spree.taxon_rules`); apps/plugins append their own.
- **Job** — `Spree::Collections::RegenerateTimeBasedJob < Spree::BaseJob` (`queue_as Spree.queues.collections`): selects automatic collections that have a rule whose `type` is registered as time-based and calls `Spree::Collections::RegenerateProducts` on each.
- **Cadence** — not hardcoded in the gem; wired by the host app's scheduler (sidekiq-cron / solid_queue recurring), the same pattern as `Spree::StockReservations::ExpireJob`. Hourly/daily is enough (`AvailableOn` windows are day-granular).

> **`AvailableOn` ships fully now, but its implementation is interim.** The STI class ships in Phase 1 (registered in `collection_rules` alongside `Tag`/`Sale`) so the Phase 4 rake migrates `TaxonRules::AvailableOn` → `CollectionRules::AvailableOn` **1:1** (value + match_policy preserved) rather than dropping the data. Its `apply` is the **same legacy `spree_products.available_on` query the taxon rule already runs** — a faithful, zero-behavior-change carry-over (the taxon rule was equally channel-unaware), which keeps migrated automatic collections computing correct membership and lets the scheduled job above refresh them. One caveat holds until the reimplementation (Migration Phase 5):
> - **Channel-unaware + on a doomed column.** The `available_on` column is superseded by the channel-scoped `ProductPublication#published_at` and is **removed** by `6.0-channels-catalogs-b2b.md` (its Phase 3 — "6.0 cleanup"). So `AvailableOn` is a **new reader of a column another 6.0 plan drops**. **Migration Phase 5** reimplements `apply` against `published_at` and — as a hard cross-plan ordering constraint — must land **before** the channels plan drops the column.

### Product changes

**Already shipped:** `has_many :categories, through: :classifications` and a `category_ids=` setter that maps the API name to the underlying `taxon_ids` (accepting prefixed IDs, scoped to the product's store); `main_taxon` still exists. Still to add: `primary_category`, the `collections` association, and the `collections_count` counter. **To remove:** `brand_taxon` / `brand_name` (brands out of scope — see Key Decisions → Brands).

Unlike categories (a nested set that needs an explicit `reposition` with a target parent + index), collections are a flat `acts_as_list`, so the collection's own order is reordered by sending `position` on `PATCH /collections/:id` — matching the SPA `<ResourceTable reorder>` convention and the sibling flat resource (`payment_methods`). Only the **nested** membership (`ProductCollection.position`) keeps a dedicated `reposition`. Admin collection endpoints require the new `read_collections` / `write_collections` API-key scopes (JWT admins use CanCanCan as usual).

Nested `rules` are written through a model-level **sync setter** (`Spree::Collection#rules=`), mirroring `Spree::OptionType#option_values=`: the client sends the full desired rule set under `rules` (prefixed `crule_` ids update, id-less entries create, omitted rules are destroyed), decoded and applied in the parent's save transaction. `normalize_params` is deliberately not called on this controller so `rules` reaches the setter rather than being rewritten to `rules_attributes`.

Collection response (**Admin** — the Store serializer omits the merchandising config `automatic`, `rules_match_policy`, and `rules`; that's back-office state, not customer-facing):
```json
{
  "id": "coll_k5nR8xLq",
  "name": "Summer Sale",
  "automatic": true,
  "sort_order": "best_selling",
  "rules_match_policy": "all",
  "rules": [
    { "id": "crule_abc", "type": "Spree::CollectionRules::Tag", "value": "summer", "match_policy": "contains" }
  ],
  "image_url": "https://...",
  "products_count": 42,
  "slug": "summer-sale"
}
```
The **Store** response is the customer-facing subset — `id`, `name`, `slug`/`permalink`, `description`/`description_html`, `image_url`/`square_image_url`, `sort_order`, `hide_from_nav`, `position`, `meta_*`, `products_count` (+ `custom_fields` on expand) — with **no** `automatic`/`rules_match_policy`/`rules` (internal merchandising state stays out of the Store API, per the store-vs-admin serializer rule).

### Collection product listing (Store API)

Products in a collection are served by a **dedicated endpoint**, not a filter on `/products`. This follows the Shopify Storefront GraphQL model (`collection { products }`) where the collection is the path context and filters/sort layer on top. (Most REST APIs — WooCommerce, Medusa, commercetools, BigCommerce — use a `?collection_id=` filter instead; we deliberately choose the nested style for cleaner requests and to apply the collection's own default sort automatically. `in_collection` is still kept as a Ransack scope for composite queries.)

- `GET /api/v3/store/collections/:id/products` — a thin controller that resolves the collection and **delegates to the same search-provider flow** as `/products` (query, Ransack filters, pagination, serializer). `def scope` stays the **base visibility scope** (`available` + `accessible_by`) — it is **not** pre-scoped to the collection. Instead the controller injects `in_collection: collection.prefixed_id` into the filters, and each provider applies it: the DB provider via the `in_collection` Ransack scope, Meilisearch via the `collection_ids` filter. Collection scoping has to be a provider filter (not an AR pre-scope) because Meilisearch does its own filtering/counting/pagination and treats the AR scope only as a post-search visibility guard. No logic duplication.
- **Sort precedence.** Effective sort = the request's `sort` param if present, otherwise the collection's `sort_order`. Position ordering (`ProductCollection.position`) applies **only when the effective sort is `manual`**; any dynamic sort — whether from the collection's own default or a shopper override — dismisses it. Because the listing runs through the provider (for the `in_collection` filter, pagination, and faceting), manual sort **cannot** be handled by bypassing the provider with a direct AR query — it must be implemented *in* the provider. That's the same provider-level manual-sort fix categories need, which is **unresolved** (Open Question #1, approach TBD); until it lands, a `sort_order: 'manual'` collection falls back to default order. The dedicated endpoint does **not** sidestep the v3 manual-sort gap.
- **Facets/sort metadata** reuse the existing `GET /api/v3/store/products/filters` with a new **`collection_id`** param (parallel to `category_id`): it resolves the collection, scopes `in_collection`, and passes **only the collection's `sort_order`** to `FiltersAggregator` — via a `sort_order:` string param, **not** the `Collection` object (the aggregator stays decoupled from the model). No child facet (collections are flat).
- **Model:** `Spree::Product.in_collection` scope (flat join to `spree_product_collections`, no descendants), whitelisted as a ransackable scope. (No `in_collections` plural — the collection surface is single-collection; add it only if a real multi-collection filter appears.)
- **Meilisearch:** index `collection_ids` (flat, no ancestors) on the product document, add it to `filterable_attributes` / `facet_attributes`, and handle `in_collection` in `build_filter_condition`. Kept fresh by `RegenerateProducts` (touches products + `enqueue_search_index`).

Membership is materialized in `ProductCollection` for both manual and automatic collections, so read-time is always a plain join filter — rules are never evaluated per request.

**Additional Phase 2 decisions (implemented):**

- **`classification_count` is dropped now** (not deferred). The direct taxon-side Rails counter was only read by the legacy `spree/admin` engine; the descendant-inclusive `products_count` is kept. The product-side counter is renamed `classification_count` → `categories_count`. `Classification`'s counter_cache follows to `categories_count`; the three taxon bulk services recompute `products_count` via `recalculate_products_count` instead of the removed counter. Legacy `spree/admin` views that read `taxon.classification_count` are **not** fixed (that engine is being replaced by the SPA) — the specs they break are marked `pending`.
- **Mobility translation table is pinned transitionally.** Setting `Spree::Taxon.table_name = 'spree_categories'` makes Mobility derive `spree_category_translations` / `spree_category_id`, but that table is still `spree_taxon_translations` (renamed in 6.1). So the `translates` call pins `table_name: 'spree_taxon_translations', foreign_key: 'spree_taxon_id'` until the 6.1 sweep.
- **Automatic-taxon matching code is removed here** (see Phase 3 rationale). Because 6.0 is single-delivery, the taxon auto-match path doesn't need to survive the rename: deleted `Products::AutoMatchTaxons` (+ job), `Taxons::RegenerateProducts`, `Product#auto_match_taxons` + `after_commit`, `Product.bulk_auto_match_taxons`, `Taxon#regenerate_taxon_products` + `after_commit`, and the `TaxonRule` regeneration callback. Rule-based membership now lives on `Collection`. `Price` re-match + the bulk-tags API endpoint now trigger `auto_match_collections` / `Product.bulk_auto_match_collections`; the bulk-category endpoints no longer re-match (category membership never affected rule matching).
- **`eligible_for_taxon_matching?` → `eligible_for_collection_matching?`** (the `Product` predicate + the `Price` attribute).

### Phase 3: Class renames + deprecation aliases

Now that Collections exist (Phase 1) and the tables are renamed (Phase 2), rename the remaining classes and drop the transitional `self.table_name` tweaks from Phase 2. **This runs before the data migration (Phase 4)** so the migration script is written against the final model names (`Spree::Category`, `Spree::ProductCategory`, `Spree::Collection`) rather than the transitional ones. The inheritance is inverted today — `Spree::Category < Spree::Taxon`; the rename **flips it**: `Spree::Category` becomes the base class on `spree_categories`, and `Spree::Taxon` becomes the deprecation alias (removed in 6.1).

- `Spree::Taxon` → `Spree::Category` (keep `Spree::Taxon` as a deprecation alias of `Spree::Category` for one release)
- `Spree::Taxonomy` → deprecated, no replacement
- `Spree::Classification` → `Spree::ProductCategory` (alias for one release)
- `Spree::PromotionRuleTaxon` → `Spree::PromotionRuleCategory`; `Spree::Promotion::Rules::Taxon` → `Spree::Promotion::Rules::Category` (wire format is already `category` — `api_type` / `category_ids` — so the API contract is unchanged)
- **Automatic-taxon matching code was removed in Phase 2, not here** — since 6.0 ships all phases at once (no released intermediate state), the taxon auto-match services + triggers were deleted in Phase 2 and rule-based membership moved to `Collection`. `Spree::TaxonRule` / `TaxonRules::*` remain as **data-only** models.
- **Dropping `spree_taxon_rules` + `Spree::TaxonRule` / `TaxonRules::*` + the `config.spree.taxon_rules` registry is deferred to 6.1 (Phase 6), NOT 6.0.** The Phase 4 data migration *reads* `spree_taxon_rules` to build `CollectionRules`, but schema migrations (`db:migrate`) run automatically at deploy while the data migration is a manual rake task run afterward — a same-release `drop_table` would wipe the source before the data task runs. So the table + models are retained through 6.0 and dropped in 6.1.
- The Mobility translation-table pin from Phase 2 carries over to `Spree::Category`; the actual table rename (`spree_taxon_translations` → `spree_category_translations` + FK) and dropping the pin are 6.1 (Phase 6).
- Downstream shim: `alias_attribute :classification_count, :categories_count` on Product for the deprecation window (any app reading the old column keeps working)
- Update all serializers, controllers, services, factories, specs for the rename

### Phase 4: Data migration (rake task)

Runs after the Phase 2 rename **and the Phase 3 class rename** — so the script reads via the final models (`Spree::Category`, `Spree::ProductCategory`) and writes the final Collection classes, with everything in place. Classify by **structure, not taxonomy name.** Taxonomy names are Mobility-translated/localized and freely renamed, and default taxonomies are no longer auto-seeded (removed in 5.5, PR #14020) — so name matching is unreliable across upgraded and new stores alike. The real Category/Collection signal is already in the data: whether a category is rule-based.

```ruby
# rake spree:migrate_taxons_to_categories_and_collections

# For every store, using store_id (backfill first — spree:taxons:backfill_store_id):
```

## Constraints on Current Work

- **Use `categories` in all new API code.** The Store API already does this. Don't introduce new `taxon` references in API v3.
- **Use `Spree::Category` alias in new code where possible.** It exists today as `< Taxon`.
- **Don't add new TaxonRule subtypes.** `TaxonRule` is data-only from 6.0 (matching removed); new rule types go on `CollectionRule`.
- **Don't add new Taxonomy-dependent features.** Taxonomy is going away.
- **Keep the `ctg_` prefix.** Already in use, will carry over.
- **New "collection-like" (rule-based) features go on `Spree::Collection`** — the taxon automatic-match path was removed in Phase 2.
- **`ProductTypeCategory` join table** (from 6.0-product-types.md) should be named anticipating the Category rename — use `category_id` column, not `taxon_id`. (Already satisfied: `6.0-product-types.md` specifies `category_id`.)
- **Counter reads use the new names.** New code reads `products_count` (inclusive) on Category / `categories_count` on Product — not `classification_count` (being dropped/renamed).
- **No brand-specific features.** Brands are out of scope for 6.0 (see Key Decisions → Brands). Don't add a `Spree::Brand`, a brand column, a `:brands` tag context, or a category `kind` column; model brand grouping as a Category or Collection.

## Open Questions

> **Resolved:**
> - *Brand handling* & *Category kind/type column* — 6.0 ships no brand feature and no `kind` column; brands are modeled as a regular Category or Collection, and `brand_taxon`/`brand_name` are removed with `Taxonomy`. See Key Decisions → Brands.
> - *Collection product listing* — a dedicated `GET /collections/:id/products` (base visibility scope + an injected `in_collection` filter, delegating to the shared search flow; default sort from the collection's `sort_order`), with facets via the existing `/products/filters` + a new `collection_id` param. See API → Collection product listing.
> - *Automatic collection scheduling* — a `RegenerateTimeBasedJob` scoped to a `time_based_collection_rules` registry, scheduled by the host app (the `ExpireJob` pattern). Ships in Phase 1 with `AvailableOn` as its one registered rule (interim legacy-column implementation; channel-aware reimplementation in Phase 5). See Design Details → Automatic collection regeneration.
> - *Collection nesting* — collections stay **flat**; hierarchy lives in Category (matches Shopify and keeps the Category/Collection split clean).

1. **"Manual" sort in the v3 Store API (categories *and* collections).** Manual/position ordering isn't implemented in the v3 search-provider path (see Current State → Known gap) — only in the admin controller and the legacy `Products::Find` finder. Both category pages and the collection endpoint list through the search provider, so neither can bypass it; the fix is provider-level and shared. Collections can ship without it (dynamic sorts all work; a `sort_order: 'manual'` collection just falls back to default order until this lands). The open part is the exact approach:
   - **DB provider:** add a `'manual'` branch to `SearchProvider::Database#apply_sort` and thread the category/collection context into `search_and_filter` / `apply_sort` (they don't receive it today). For a **category** this orders by the grouping's min join-position (`ascend_by_taxons_min_position` — the `MIN`/`GROUP BY` collapses a product that sits under a category *and its descendants*). A **collection** needs no such scope: it's flat and a PLP targets one collection with a unique `(collection_id, product_id)`, so it's a plain `joins(:product_collections).where(collection_id:).order('spree_product_collections.position ASC')`.
   - **Meilisearch:** fall back to the DB for the manual case (resolve ordered ids via the min-position scope, bypassing MS sort); native per-grouping position sort isn't feasible (position is a many-valued join property; MS sorts only on fixed scalar attributes).
