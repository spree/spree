# Split Adjustments into Typed Tables

**Status:** Draft
**Target:** Spree 6.0
**Depends on:** None (standalone)
**Author:** Damian + Claude
**Last updated:** 2026-07-22

## Summary

Replace the polymorphic `Spree::Adjustment` model with three typed models: `TaxLine`, `DiscountLine`, and `Fee`. Each has concrete foreign keys to its source (no polymorphic `source_type`/`source_id`), enabling eager loading and eliminating the N+1 queries that make the current adjustment system slow.

The `Spree.adjusters` extensibility pattern is preserved — developers register custom adjusters that write to the appropriate typed table.

## Problem

### 1. Polymorphic source prevents eager loading

`Adjustment` has `belongs_to :source, polymorphic: true` where source can be `TaxRate`, `PromotionAction`, or `ReturnAuthorization`. Rails cannot `includes(:source)` on a polymorphic association — every `adjustment.source` triggers a lazy load. With 12-15 adjustments per order, each recalculation does 12-15 individual source queries.

### 2. Polymorphic adjustable adds complexity

`belongs_to :adjustable, polymorphic: true` (LineItem or Shipment) means you can't do a simple JOIN from line items to their adjustments without going through `all_adjustments` on the order.

### 3. One table does too many jobs

The same `spree_adjustments` row represents:
- Tax (from TaxRate, may be included or additional)
- Promotions (from PromotionAction, competes with other promos)
- Shipping discounts (from FreeShipping PromotionAction)
- Manual adjustments (no source)
- Return credits (from ReturnAuthorization)

Scopes like `tax`, `promotion`, `shipping`, `price` all filter by string-matching `source_type` or `adjustable_type` — unindexed class name comparisons.

### 4. Recalculation is N+1 heavy

`AdjustmentsUpdater` iterates each adjustment and calls `adjustment.update!` → `source.compute_amount(target)` → loads the source → for promotions, loads `source.promotion` → checks `promotion.eligible?(target)`. Each step is a lazy load.

### 5. State machine is unnecessary

The open/closed state machine exists to freeze adjustments on completed orders. But order completion already locks the order — per-adjustment state is redundant bookkeeping.

### Current blast radius

## Key Decisions (do not deviate without discussion)

- **Three typed models** replace one polymorphic model:
  - `Spree::TaxLine` — tax charges, concrete FK to `tax_rate_id`
  - `Spree::DiscountLine` — promotions and manual discounts, concrete FK to `promotion_action_id` (nullable for manual). Named `DiscountLine` (parallel to `TaxLine`), **not** `Discount` — "discounts" remains the applied-promotions surface backed by `OrderPromotion` (see Resolved Question 10)
  - `Spree::Fee` — extensible charges (custom fees, surcharges, future use), no polymorphic source

- **Concrete FKs to line_item and fulfillment** (not polymorphic adjustable). TaxLines and DiscountLines belong directly to either a `line_item` or a `fulfillment` via separate nullable FKs. This enables simple `includes(:tax_lines)` on LineItem.

- **The fulfillment-side FK is named `fulfillment_id` from day one**, targeting `spree_shipments` via `belongs_to :fulfillment, class_name: 'Spree::Shipment'`. These are brand-new 6.0 tables; API v3 already exposes shipments as fulfillments, and `6.0-fulfillment-and-delivery.md` renames the model in the same release — naming the column `shipment_id` now would force a rename migration of freshly created tables. When the model rename lands, only the `class_name` string changes.

- **All three models belong to `order`** directly — for order-level queries (`order.tax_lines`, `order.discount_lines`).

- **Remove the state machine.** No open/closed states. Adjustments on completed orders are immutable by convention (enforced at the service layer, not per-row state).

- **Remove `eligible` flag.** The "best promo wins" logic writes only the winning discount. Losers are not stored as ineligible rows — they simply don't exist. Simpler data, fewer writes.

- **Discount lines are strictly negative; no zero-amount placeholders.** `DiscountLine` validates `amount < 0` (`Fee` mirrors it with `>= 0`; positive manual charges are `Fee`s). `FreeShipping`'s zero-persist placeholder is not ported — the trigger gap it papered over is closed instead: selecting a delivery rate re-runs free-shipping activation (see "Promotion application flow"). Promotion usage counts only actual applications (`DiscountLine` rows), not `OrderPromotion` connections.

- **Keep `Spree.adjusters` registry** for extensibility. Custom adjusters write to `Fee` (or `DiscountLine` for custom discount types). Registration pattern unchanged.

- **New adjusters live in `Spree::Adjusters::*`** (`Base`, `Promotion`, `Tax`); the legacy `Spree::Adjustable` namespace is removed **entirely** — it's named after the polymorphic association this plan deletes, and only the adjusters would survive in it. The adjuster contract changes anyway (write typed rows; no `@totals` hash), so reusing the old class names would let old-contract custom adjusters keep loading while silently no longer counting toward totals. With a new namespace they fail loudly at boot instead — call this out in the 6.0 upgrade guide.

- **Denormalized totals on LineItem/Shipment/Order stay.** The `included_tax_total`, `additional_tax_total`, `promo_total`, `adjustment_total` columns remain — they're the performance optimization. But they're now computed from typed tables, not from polymorphic filter scopes.

- **`taxable_adjustment_total` / `non_taxable_adjustment_total` go.** Their only consumer is the adjuster `@totals` bookkeeping being replaced, and since #14295 the tax basis is computed live (`taxable_basis`), not from these columns. 6.0 stops writing them; the physical column drop is 6.1 cleanup (they shipped, so they get the deprecation window — same treatment as `is_master`).

- **Recalculation order is fixed: discounts → fees → tax — enforced by adjuster-declared `type`, not registry position.** Tax must run last because it's computed from the discounted basis (#14295); today's `AdjustmentsUpdater` enforces that with a class-name string match on the tax adjuster. Instead, every adjuster declares its `type` on the class (`:discount` / `:fee` / `:tax`, default `:fee`), and `OrderUpdater` runs the registry partitioned by type. Registration order only matters within a type, so `Spree.adjusters << MyAdjuster` is always order-safe.

## Design Details

### TaxLine

```ruby
class Spree::TaxLine < Spree.base_class
  has_prefix_id :tl

  belongs_to :order, class_name: 'Spree::Order', inverse_of: :tax_lines
  belongs_to :tax_rate, class_name: 'Spree::TaxRate'

  # Concrete adjustable — one of these is set, not both.
  # class_name flips to 'Spree::Fulfillment' when 6.0-fulfillment-and-delivery lands — no schema change.
  belongs_to :line_item, class_name: 'Spree::LineItem', optional: true
  belongs_to :fulfillment, class_name: 'Spree::Shipment', optional: true

  validates :amount, numericality: true
  validates :label, presence: true
  validate :exactly_one_adjustable

  # Replaces adjustment.included? — tax can be included in price or additional
  attribute :included, :boolean, default: false

  scope :included_in_price, -> { where(included: true) }
  scope :additional, -> { where(included: false) }
  scope :for_line_items, -> { where.not(line_item_id: nil) }
  scope :for_fulfillments, -> { where.not(fulfillment_id: nil) }

  def adjustable
    line_item || fulfillment
  end

  private

  def exactly_one_adjustable
    if line_item_id.blank? && fulfillment_id.blank?
      errors.add(:base, 'must belong to a line item or fulfillment')
    elsif line_item_id.present? && fulfillment_id.present?
      errors.add(:base, 'cannot belong to both a line item and fulfillment')
    end
  end
end
```

### DiscountLine

Named `DiscountLine` (parallel to `TaxLine`) because "discount" is already the public applied-promotions concept: `Order#discounts` aliases `order_promotions`, `OrderPromotion` owns the `discount_` ID prefix, and the Store API `discounts` array serializes that join. A `DiscountLine` is where a promotion's money actually landed, line by line.

```ruby
class Spree::DiscountLine < Spree.base_class
  has_prefix_id :dl

  belongs_to :order, class_name: 'Spree::Order', inverse_of: :discount_lines
  belongs_to :promotion_action, class_name: 'Spree::PromotionAction', optional: true
  belongs_to :promotion, class_name: 'Spree::Promotion', optional: true

  # Concrete adjustable — always one of these (order-level discounts distributed to line items)
  belongs_to :line_item, class_name: 'Spree::LineItem', optional: true
  belongs_to :fulfillment, class_name: 'Spree::Shipment', optional: true

  # A discount is strictly a credit. Zero results are never written (no
  # free-shipping placeholders — see "Promotion application flow"); positive
  # manual charges are Fees.
  validates :amount, numericality: { less_than: 0 }
  validates :label, presence: true
  validate :exactly_one_adjustable

  scope :for_line_items, -> { where.not(line_item_id: nil) }
  scope :for_fulfillments, -> { where.not(fulfillment_id: nil) }
  # Promotion#kind is an integer enum (coupon_code: 0, automatic: 1) — merge the enum scope,
  # don't string-match the column
  scope :automatic, -> { joins(:promotion).merge(Spree::Promotion.automatic) }
  scope :manual, -> { where(promotion_action_id: nil) }

  def adjustable
    line_item || fulfillment
  end

  def promotion?
    promotion_action_id.present?
  end

  def manual?
    promotion_action_id.nil?
  end
end
```

### Fee (extensible)

```ruby
class Spree::Fee < Spree.base_class

  belongs_to :order, class_name: 'Spree::Order', inverse_of: :fees

  # Concrete adjustable
  belongs_to :line_item, class_name: 'Spree::LineItem', optional: true

  validates :amount, numericality: { greater_than_or_equal_to: 0 }
  validates :label, presence: true
  validates :kind, presence: true  # e.g., 'surcharge', 'handling', 'gift_wrap'

  scope :for_line_items, -> { where.not(line_item_id: nil) }
  scope :for_fulfillments, -> { where.not(fulfillment_id: nil) }

    line_item || fulfillment
  end

```

**Fee is strictly a buyer-facing charge** — it rolls into `order.fee_total` and the total the customer pays. Marketplace commission is **not** a Fee: it's the off-order `Spree::CommissionLine` ledger (see `6.0-multi-vendor-marketplace.md`, Decision 2). Do not add a commission/marketplace `kind`.

### Recalculation (new)

`AdjustmentsUpdater` is removed; `OrderUpdater` absorbs it as one linear pipeline. The `Adjustment` `after_create`/`after_destroy` recalculation callbacks disappear with the model — recalculation runs only where `update_with_updater!` is invoked, never as a side effect of row writes. The checkout `create_tax_charge!` / `create_shipment_tax_charge!` hooks are removed with them: every checkout transition and shipment-cost write already ends in a pipeline run, so the explicit tax calls were pure duplication.

Completed orders are never recalculated (`return if order.completed?`) — their adjustment lines are immutable by convention, replacing the old open/closed adjustment state machine.

```ruby
class Spree::OrderUpdater
  def update_adjustment_total
    recalculate_adjustments

    # Concrete FKs make these two grouped aggregates over the denormalized
    # per-adjustable columns — no N+1
    order.adjustment_total = <line item + shipment adjustment_total sums>
    order.included_tax_total = ...; order.additional_tax_total = ...; order.promo_total = ...
    order.fee_total = Spree::Fee.where(order_id: order.id).sum(:amount)

    update_order_total
  end

  private

  # The pipeline that replaces AdjustmentsUpdater. Adjusters run partitioned by
  # their declared type — discounts, then fees, then tax last (tax reads the
  # discounted basis, #14295). Registration order only matters within a type,
  # with one constraint inside :discount — see "Best-promo selection".
  def recalculate_adjustments
    return if order.completed?

    # ONE batched load for the whole order — possible only because the FKs are
    # concrete (the polymorphic source never was preloadable). Adjusters work
    # on these loaded associations in memory; they must not re-query per
    # adjustable (no .where/.reload on the association — that discards the preload).
    adjustables = order.line_items.includes(:fees, discount_lines: [:promotion, :promotion_action]).to_a +
                  order.shipments.includes(:fees, discount_lines: [:promotion, :promotion_action]).to_a

    discount_adjusters, fee_adjusters, tax_adjusters =
      Spree.adjusters.group_by(&:type).values_at(:discount, :fee, :tax)

    # adjust_all is the adjuster entry point: per-adjustable adjusters iterate
    # (new(order, adjustable).update); the tax and order-level-promotion
    # adjusters override it to work on the whole batch at once.
    [*discount_adjusters, *fee_adjusters, *tax_adjusters].each do |adjuster|
      adjuster.adjust_all(order, adjustables)
    end

    # Interim: Adjusters::Tax delegates to TaxRate.adjust(order, adjustables) once;
    # 6.0-tax-provider.md later swaps its internals for the provider call.

    persist_adjustable_totals(adjustables)
  end

  # Rolls the typed lines up into each adjustable's denormalized total columns:
  # four grouped aggregate queries for the whole order (discounts, fees,
  # included tax, additional tax keyed by [line_item_id, fulfillment_id]),
  # then per-adjustable update_columns, written only on change.
  def persist_adjustable_totals(adjustables)
    ...
  end
end
```

`LineItem#taxable_basis` / `Shipment#taxable_basis` compute from the **live** `discount_lines` association (`[amount + current_discount_total, 0].max`, skipping destroyed rows), not the denormalized `promo_total` column — the tax pass runs inside the pipeline *before* the columns are persisted, so the column is one run stale at that point.

`taxable_adjustment_total` / `non_taxable_adjustment_total` are no longer written (see Key Decisions). `LineItem#taxable_basis` and `Order#order_level_promo_total` (added by #14295) are reimplemented on top of the typed tables — with order-level discounts materialized as distributed `DiscountLine` rows, the live proportional-share computation collapses to `line_item.discount_lines.sum(:amount)`.

### TaxRate (source side changes — interim)

**This shape is interim.** `6.0-tax-provider.md` depends on this plan for the `TaxLine` model and then removes `TaxRate.adjust` entirely — the provider becomes the only tax entry point and writes `TaxLine` records itself (its Internal provider already sketches the same `upsert_tax_line`). Here, `TaxRate.adjust` switches from destroy-and-recreate `Adjustment` rows to upserting `TaxLine` rows; amounts keep the discounted-basis semantics from #14295 (`compute_amount` reads `item.taxable_basis`, not raw amount).

```ruby
class Spree::TaxRate < Spree.base_class
  # Remove: include Spree::AdjustmentSource
  # No dependent: option — tax_rate_id is null: false, so :nullify is impossible.
  # TaxRate is paranoid; TaxLine's belongs_to uses -> { with_deleted } so frozen
  # lines on completed orders keep resolving after a rate is soft-deleted.
  has_many :tax_lines, class_name: 'Spree::TaxLine'

  def self.adjust(order, items)
    return if items.none?

    # NO early return when no rates match — items that lost their rates (zone
    # change, category change, deleted rate) must still shed stale lines.
    rates = match(order.tax_zone)

    # Items whose tax_category matches a rate: store_pre_tax_amount, destroy
    # lines whose rate no longer applies, then upsert one line per applicable
    # rate (the instance-level rate.adjust destroys the line instead when the
    # amount is zero — tax lines only exist while tax is owed).
    #
    # Items without any applicable rate: store_pre_tax_amount(item, []) then
    # tax_lines.destroy_all — pre_tax_amount is written for untaxed items too
    # (legacy left it 0 unless the item was previously taxed); refunds read it.
  end
end
```

`store_pre_tax_amount` writes from the same **live** basis tax is computed on: `item.taxable_basis` (amount net of ALL discounts — item-level and distributed whole-order shares — clamped at zero), with included taxes divided out. Not `discounted_amount`, which reads the stale `promo_total` column mid-pipeline. `pre_tax_amount` therefore means "what the customer actually pays before tax", and refund math reads it directly with no separate weighted order-adjustment share.

### PromotionAction (source side changes)

Uses the `Spree::PromotionActions::*` namespace per the STI-namespacing decision (`decisions.md` 2026-03-16). That rename has **not landed yet** — the code today is still `Spree::Promotion::Actions::*`; whichever plan ships first carries the rename.

```ruby
class Spree::PromotionAction < Spree.base_class
  # Remove: include Spree::AdjustmentSource
  has_many :discount_lines, class_name: 'Spree::DiscountLine', foreign_key: :promotion_action_id

  # Port of AdjustmentSource#deals_with_adjustments_for_deleted_source:
  # destroy lines on incomplete orders; completed orders keep frozen rows
  # (PromotionAction is paranoid; DiscountLine's belongs_to is with_deleted).
  before_destroy :destroy_discount_lines_on_incomplete_orders

  # Whether perform writes an order-wide distributed group (CreateAdjustment)
  # rather than independent per-adjustable lines. Drives which discount
  # adjuster maintains the rows — see "Best-promo selection".
  def order_level? = false

  protected

  # The single write path all actions share. A non-negative amount destroys
  # the existing line instead (never leaving an unsaved ghost in the loaded
  # association) — DiscountLine amounts are strictly negative.
  def upsert_discount_line(order, adjustable, amount)
    ...
  end
end

class Spree::PromotionActions::CreateItemAdjustments < Spree::PromotionAction
  def perform(payload = {})
    # Writes candidate discount lines; best-promo selection happens in the
    # adjuster. The negative-total clamp reads only the surviving order-level
    # lines (CreateAdjustment groups) — item-level and fulfillment lines never
    # feed the clamp.
    order.line_items.each do |line_item|
      upsert_discount_line(order, line_item, compute_amount(line_item))
    end
  end
end

class Spree::PromotionActions::CreateAdjustment < Spree::PromotionAction
  def order_level? = true

  # Distributes the whole-order amount across line items by largest-remainder:
  # shares weighted by line_item.amount / order.item_total, truncated toward
  # zero, the rounding deficit assigned to the lowest-id line — deterministic,
  # so recalculation reproduces perform's split exactly. distributed_amounts
  # returns the full {line_item_id => share} map in ONE pass per order;
  # compute_amount(line_item) reads that line's share out of it.
  def perform(payload = {})
    shares = distributed_amounts(order)
    order.line_items.each do |line_item|
      upsert_discount_line(order, line_item, shares.fetch(line_item.id, 0))
    end
  end
end
```

### Promotion application flow (activation creates, recalculation maintains)

The division of labor stays as today, with one gap closed:

- **Activation creates.** The `PromotionHandler`s keep their roles — `Coupon` connects the promotion to the order (`OrderPromotion`) and performs actions; `Cart` re-performs the connected + automatic promotions on every cart mutation. `action.perform` is the only creator of promotion-backed `DiscountLine` rows, and zero results are never written.
- **Recalculation maintains.** `Adjusters::Promotion` recomputes existing rows, deletes rows that zero out or lose eligibility, and enforces best-promo-per-adjustable. It never creates rows. Manual discount lines (`promotion_action_id: nil`) are never touched by the adjuster.
- **New activation trigger.** Selecting a delivery rate re-runs free-shipping activation for the order before recalculating. This closes the one gap in today's event coverage: every line-item input change is a cart mutation (which activates), but rate selection (`shipment.selected_shipping_rate_id=` → `ShippingRate` callback) only recalculates — which is why today's `FreeShipping#create_adjustment` persists a $0 placeholder for recalculation to update. With the trigger in place, the placeholder and the override are not ported. Mechanism: in the `selected_shipping_rate_id=` setter itself (the Store API fulfillments endpoint assigns the attribute directly, so a service-level trigger would miss it), after the `order.completed?` guard — write the new cost, reset the order's shipments association so activation reads fresh costs, `PromotionHandler::FreeShipping.new(order).activate`, then `order.set_shipments_cost` to run the pipeline.
- **Coupon application recalculates before judging the result.** `PromotionHandler::Coupon` runs `order.update_with_updater!` after `promotion.activate` and *before* checking which discount lines exist — best-promo competition prunes losing candidates in that run, so a coupon whose lines lost reports `:coupon_code_better_exists` instead of a false success. One full updater run per apply (the legacy flow got its equivalent recalculation implicitly via `Adjustment` callbacks).
- **Usage counting follows actual application.** `Promotion#credits_count` / `usage_limit_exceeded?` count distinct orders with `DiscountLine` rows from the promotion's actions. A promotion that is connected (`OrderPromotion`) but produced no discount — a free-shipping coupon on an order whose shipping costs $0 — is not "used". The customer-facing applied-promotions display is unaffected: the Store API `discounts` array serializes the `OrderPromotion` join, not `DiscountLine` rows (Resolved Question 10).

### Best-promo selection (simplified)

Instead of marking losers as `eligible: false`, recalculation refreshes the candidates that activation wrote and keeps only the winner. The discount pass is **two adjusters**, because the two competition scopes differ (see Resolved Question 11): `Adjusters::OrderLevelPromotion` maintains the order-wide distributed groups written by `order_level?` actions, and `Adjusters::Promotion` runs per-adjustable competition for everything else. OrderLevelPromotion **must be registered before** Promotion — the item-level negative-total clamp reads the surviving order-level lines.

```ruby
# Every adjuster declares which pass it runs in (and which table family it
# writes). type is a class-level declaration on POROs — no AR/STI involved.
class Spree::Adjusters::Base
  class_attribute :type, default: :fee  # :discount | :fee | :tax

  # The pipeline entry point. The default iterates per adjustable; batch
  # adjusters (Tax, OrderLevelPromotion) override it to see the whole order
  # at once.
  def self.adjust_all(order, adjustables)
    adjustables.each { |adjustable| new(order, adjustable).update }
  end

  def initialize(order, adjustable)
    @order = order
    @adjustable = adjustable
  end

  private

  attr_reader :order, :adjustable
end

class Spree::Adjusters::Promotion < Spree::Adjusters::Base
  self.type = :discount

  def update
    # Candidates were created by PromotionAction#perform at activation time —
    # recalculation refreshes them, it never creates rows. The association is
    # preloaded by OrderUpdater; filter in Ruby, don't re-query. Order-level
    # groups are excluded — OrderLevelPromotion already settled them.
    candidates = adjustable.discount_lines.select do |line|
      line.promotion? && !line.destroyed? && !line.promotion_action.order_level?
    end

    # Recompute each candidate: refresh its amount, or destroy it when it
    # zeroes out or loses eligibility (strictly-negative invariant — zeroed
    # rows are deleted, not kept).
    candidates.each do |line|
      amount = line.promotion_action.compute_amount(adjustable)

      if amount >= 0 || !line.promotion&.eligible?(adjustable)
        line.destroy!
      elsif line.amount != amount
        line.update!(amount: amount)  # write only on change
      end
    end

    candidates = candidates.reject(&:destroyed?)

    # Best promo wins per adjustable: most negative amount, ties broken by
    # newest, then by highest id (created_at has second precision — the id
    # key keeps same-second activations deterministic).
    best = candidates.min_by { |line| [line.amount, -line.created_at.to_i, -line.id] }
    return unless best

    (candidates - [best]).each(&:destroy!)
  end
end

class Spree::Adjusters::OrderLevelPromotion < Spree::Adjusters::Base
  self.type = :discount

  # Order-wide group maintenance for order_level? actions: groups the loaded
  # lines by action, checks eligibility once per order, refreshes every group
  # from ONE distributed_amounts batch per action (the standing constraint:
  # never compute the distribution more than once per order per run), then
  # keeps the best group — by total distributed amount, ties by newest, then
  # highest id — and destroys the others whole. Groups win or lose atomically,
  # which is what preserves today's stacking behavior.
  def self.adjust_all(order, adjustables)
    ...
  end
end
```

No `eligible` flag, no write-then-update-all. At rest, only the winners (plus any manual discount lines) are stored; competing candidates recreated by the next activation are pruned again in the same pass. The `Spree.adjusters` registry default is `[Adjusters::OrderLevelPromotion, Adjusters::Promotion, Adjusters::Tax]`.

The three models share a `Spree::AdjustmentLine` concern (the adjustable `belongs_to`s, the exactly-one-adjustable validation — checking the **associations**, not the `_id` columns, so built-not-yet-saved records validate — the `adjustable` resolver, label presence, `for_line_items`/`for_fulfillments` scopes, `DisplayMoney`).

### Extensibility

Custom adjustment types register as adjusters and write to `Fee`:

```ruby
# config/initializers/spree.rb
Spree.adjusters << MyApp::Adjusters::GiftWrap

# app/models/my_app/adjusters/gift_wrap.rb
class MyApp::Adjusters::GiftWrap < Spree::Adjusters::Base
  # type defaults to :fee — no declaration needed

  def update
    return unless adjustable.respond_to?(:gift_wrap?) && adjustable.gift_wrap?

    adjustable.fees.find_or_initialize_by(kind: 'gift_wrap', order: adjustable.order).tap do |fee|
      fee.amount = 5.99
      fee.label = 'Gift wrapping'
      fee.save!
    end
  end
end
```

For custom discount types, same pattern with `DiscountLine`:

```ruby
class MyApp::Adjusters::LoyaltyDiscount < Spree::Adjusters::Base
  self.type = :discount  # runs in the discount pass, before fees and tax

  def update
    points = adjustable.order.customer&.loyalty_points || 0  # customer_id rename, decisions.md 2026-03-16
    return if points.zero?

    discount_amount = [points * 0.01, adjustable.amount * 0.1].min
    return if discount_amount.zero?  # DiscountLine validates amount < 0 — zero rows are never written

    adjustable.discount_lines.find_or_initialize_by(kind: 'loyalty', order: adjustable.order).tap do |disc|
      disc.amount = -discount_amount
      disc.label = "Loyalty discount (#{points} pts)"
      disc.save!
    end
  end
```

### Phase 3: Data migration (rake task)

Runs once at upgrade time, after the 6.0 code is deployed (same pattern as `spree:remove_master_variant`) — it backfills historical orders' typed rows from `spree_adjustments`, which stays in place until 6.1. In-flight orders don't need it: their first recalculation rewrites typed rows from source. The task must be idempotent — skip orders that already have typed rows.

Two constraints: the order-level distribution must reuse `CreateAdjustment`'s largest-remainder function so backfilled rows match what runtime recalculation would produce; and the task must **NOT rewrite `pre_tax_amount`** — completed orders' values were written under the semantics in force at completion, and refund math keeps reading them as-is.

## Constraints on Current Work

- **Don't add new adjustment source types.** New fee/discount types should be designed for the typed model.
- **Don't rely on `source_type` string matching.** New code should use the `tax`, `promotion` scopes which will be replaced by typed table queries.
- **Don't add new state machine transitions to Adjustment.** The state machine is being removed.
- **Keep denormalized totals on LineItem/Shipment/Order.** These stay — they're the performance optimization. (Except the `taxable_adjustment_total` / `non_taxable_adjustment_total` pair, which is being retired — don't add new readers/writers.)
- **Keep TaxLine/DiscountLine/Fee strictly line-item- or fulfillment-attached.** Marketplace order splitting depends on every line re-homing unambiguously to one child order.
- **Don't add a commission/marketplace `Fee` kind.** `Fee` is buyer-facing and rolls into order totals; commission is the off-order `Spree::CommissionLine` ledger (`6.0-multi-vendor-marketplace.md`, Decision 2).

## Resolved Questions

1. **Order-level discounts.** Order-level promotion amounts are **distributed proportionally across line items** at application time. No order-level discount records with nil FKs. Every `DiscountLine` always belongs to a `line_item` or `fulfillment`. This is the industry-standard approach — it simplifies tax calculation (tax applies to the discounted per-item amount), partial refunds (each item knows its actual discount), and accounting. The `CreateAdjustment` promotion action (order-level) distributes its amount across line items weighted by `line_item.amount / order.item_total`.

2. **Return credits.** Separate domain — return credits are **not** modeled as `DiscountLine`. Returns, exchanges, and claims will be addressed as first-class entities in a dedicated 6.0 plan (see: Claims/Exchanges as first-class entities). Return refunds flow through the `Refund` model and payment system, not through the adjustment/discount system.

3. **`fee_total` on Order.** Add `fee_total` column to `spree_orders`. Explicit and consistent with the existing `promo_total`, `shipment_total` pattern. Order total formula becomes: `total = item_total + shipment_total + fee_total + additional_tax_total + promo_total`.

4. **Fulfillment-side FK is `fulfillment_id` from day one.** New tables shouldn't be born with a column (`shipment_id`) already scheduled for rename in the same release. API v3 already serializes shipments as fulfillments (`Admin::FulfillmentSerializer`, `/fulfillments` routes, `delivery_method` keys), so the public vocabulary is settled. Interim wiring: `belongs_to :fulfillment, class_name: 'Spree::Shipment'`; the `6.0-fulfillment-and-delivery.md` rename then touches only `class_name` strings, not these tables.

5. **`taxable_adjustment_total` / `non_taxable_adjustment_total` are retired.** They exist on orders, line items, and shipments purely as adjuster `@totals` bookkeeping; since #14295 the VAT basis is computed live (`taxable_basis`), and with typed tables any taxable/non-taxable split is a direct query. Stop writing them in 6.0, drop the columns in 6.1 (Phase 5).

6. **Tax is computed from the discounted basis — typed tables preserve #14295 semantics.** PR #14295 (`LineItem#taxable_basis`, `Order#order_level_promo_total`, `Shipment#taxable_basis`) made all four discount × tax-mode combinations move the taxable basis, computing the whole-order promo share proportionally at tax time. This plan materializes exactly that proportional distribution as `DiscountLine` rows (Resolved Question 1), so `taxable_basis` collapses to `[amount + discount_lines.sum(:amount), 0].max` computed on the live association. Refund math is updated in the same pass: `pre_tax_amount` becomes net of ALL discounts and the weighted order-adjustment share disappears — see Phase 2.

7. **`DiscountLine.amount` is strictly negative; positive manual charges are `Fee`s.** Today's manual adjustment (source-less, either sign) splits by sign: credits → `DiscountLine (kind: 'manual')`, charges → `Fee (kind: 'manual')`. Enforced by model validations (`DiscountLine` `< 0`, `Fee` `>= 0`), which makes the Phase 3 sign split mandatory rather than advisory.

8. **No zero-amount placeholder discounts.** Today `FreeShipping#create_adjustment` persists $0 adjustments because delivery-rate selection triggers recalculation only, and recalculation can only update existing rows — the placeholder is the slot it updates. Line items never needed this: every event that changes an item discount's inputs is a cart mutation, and every cart mutation re-runs activation (`PromotionHandler::Cart`), which creates missing rows. The asymmetry is a missing trigger, not a domain constraint — so 6.0 adds the trigger (rate selection re-runs free-shipping activation) instead of porting the placeholder. Zero results are never written; recalculation deletes rows that zero out or lose eligibility.

9. **Promotion usage counts actual applications only.** `credits_count` / `usage_limit` consume when `DiscountLine` rows exist for the promotion's actions — not when the promotion is merely connected via `OrderPromotion`. Consequence, accepted: a free-shipping coupon on an order with $0 shipping shows as applied (the Store API `discounts` array reads the join) but does not consume usage.

10. **The typed model is `Spree::DiscountLine`, not `Spree::Discount` — "discounts" stays the applied-promotions surface.** "Discount" is already a public concept owned by the `OrderPromotion` join: `Order#discounts` aliases `order_promotions`, `OrderPromotion` has `has_prefix_id :discount` (public IDs `discount_…`), and the Store/Admin API `discounts` arrays (`DiscountSerializer` / `Admin::DiscountSerializer`, one entry per applied promotion with `code`/`name`/per-promotion `amount`) serialize that join. Reusing the name for typed rows would either break the Store API or silently flip `order.discounts`' meaning between 5.x and 6.0, and would leave the Admin API with an embedded `discounts` field (join) and a `/discounts` endpoint (typed rows) meaning different things on the same resource. `DiscountLine` (parallel to `TaxLine`) avoids all of it: the join keeps every "discount" name untouched; typed rows are `order.discount_lines` / `/orders/:id/discount_lines` / prefix `dl_`. Division of concepts: **discounts** = *which promotions* are applied; **discount_lines** = *where the money landed*, line by line. Bonus: `OrderPromotion#amount` reimplemented on winner-only `DiscountLine` rows makes "sum of `discounts[].amount` == `discount_total`" an invariant (today it can overstate — see Phase 2).

11. **Promotion stacking is preserved by scoping the two competitions separately.** Today's behavior — an order-level promotion, an item-level promotion, and free shipping can all apply to one order — survives because distributed order-level shares never compete against item-action rows on the same line item. Order-level (`order_level?`-action-backed) groups compete **order-wide as whole groups**: `Adjusters::OrderLevelPromotion` picks the best group by total distributed amount and destroys the losers' shares atomically (a group never half-wins). Item-action rows compete **per adjustable** in `Adjusters::Promotion`. Fulfillment discount lines (free shipping) compete only among themselves per shipment. Tie-breaks everywhere: most negative amount, then newest, then highest id.

12. **Manual admin adjustments stop affecting totals — no legacy bridge in the pipeline.** `OrderUpdater` sums typed tables only. The legacy Rails admin's manual-adjustment CRUD still writes `spree_adjustments`, but those rows no longer reach any total (spree/admin is being ditched; manual adjustments have no v3 API surface either — read-only in Phase 4). Upgrade-guide note.
