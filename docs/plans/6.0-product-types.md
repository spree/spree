# ProductType (Prototype → ProductType)

**Status:** Draft
**Target:** Spree 6.0
**Depends on:** Metafields system (shipped in 5.4), Taxon → Category rename (6.0)
**Author:** Damian + Claude
**Last updated:** 2026-03-26

## Summary

Rename and extend the existing `Spree::Prototype` model into `Spree::ProductType` — a persistent, schema-defining association on Product. Today Prototype is a fire-and-forget creation template (copies option types and taxons, then has no ongoing relationship with the product). ProductType becomes a **live template** that defines what option types, categories, and metafield definitions a product has, driving admin UI form generation and enabling type-aware filtering.

ProductType acts as an **authoritative template** for option types: changes to a type's option types propagate to all products of that type. Products may add extra option types beyond what the type provides, but cannot remove type-defined ones. This ensures catalog consistency (all "T-Shirts" share Color and Size) while allowing per-product flexibility (one T-shirt can add "Sleeve Length").

This closes Spree's biggest functional gap in catalog modeling, while building on existing infrastructure rather than starting from scratch.

## Problem

1. **No product schema enforcement.** All Spree products share the same shape. A merchant selling T-shirts and laptops has no way to say "T-shirts must have Size and Color; laptops must have RAM and Storage" without custom code.

2. **No admin form generation from type.** The admin UI shows the same fields for every product. Modern platforms auto-generate product editing forms based on ProductType — adding an attribute to a type instantly updates the form for all products of that type.

3. **Prototype is wasted infrastructure.** The model, join tables (`spree_option_type_prototypes`, `spree_prototype_taxons`), and creation-time callback already exist but serve only as a copy template with no ongoing relationship.

4. **Metafields are unstructured per product.** Any MetafieldDefinition can be attached to any product. There's no way to declare "products of this type should have these metafields."

## Key Decisions (do not deviate without discussion)

- **Rename only, no new table.** `Spree::Prototype` → `Spree::ProductType`, `spree_prototypes` → `spree_product_types`. Existing join tables renamed accordingly. No data loss.
- **Persistent `belongs_to`.** Product gets `product_type_id` FK (nullable — existing products don't require a type). The current `attr_accessor :prototype_id` + fire-and-forget callback is replaced by a real association.
- **ProductType defines OptionTypes, Categories, and MetafieldDefinitions.** Three M2M associations on ProductType, all carried over or added via join tables.
- **ProductType is a live template for option types.** Option types defined on a ProductType propagate to all products of that type. Adding an option type to the type adds it to all its products; removing one removes it from products (with safety checks). Products can add extra option types beyond the type's set but cannot remove type-defined ones. This is "additive override" — the type is the floor, not the ceiling.
- **MetafieldDefinition association carries `required` and `sort_order`.** The join table `spree_product_type_metafield_definitions` determines which metafield definitions a product type uses, whether they're required, and their display order. This is the schema enforcement layer.
- **Prefix ID: `pt_`.** Replaces `proto_`.
- **No shared value pools in this phase.** Metafield values remain per-resource strings (no shared AttributeValue rows). Faceted filtering via shared pools is a future enhancement tracked in Open Questions.
- **ProductType is optional.** Products without a type work exactly as today. Type enforcement is additive, not breaking.
- **ProductType is immutable on product after creation.** Changing a product's type would invalidate its metafield values. Block type changes once metafields are assigned.

## Design Details

### Models

```ruby
# Renamed from Spree::Prototype
class Spree::ProductType < Spree.base_class
  has_prefix_id :pt

  include Spree::Metadata
  include Spree::Metafields
  include Spree::TranslatableResource  # translatable name

  # Existing (renamed join tables)
  has_many :option_type_product_types, class_name: 'Spree::OptionTypeProductType', dependent: :destroy
  has_many :option_types, through: :option_type_product_types

  has_many :product_type_categories, class_name: 'Spree::ProductTypeCategory', dependent: :destroy
  has_many :categories, through: :product_type_categories  # taxons until 6.0 rename

  # New
  has_many :product_type_metafield_definitions, class_name: 'Spree::ProductTypeMetafieldDefinition',
           dependent: :destroy
  has_many :metafield_definitions, through: :product_type_metafield_definitions

  has_many :products, class_name: 'Spree::Product', dependent: :nullify

  validates :name, presence: true
end
```

```ruby
# New join table: which metafield definitions does this type require?
class Spree::ProductTypeMetafieldDefinition < Spree.base_class
  belongs_to :product_type, class_name: 'Spree::ProductType'
  belongs_to :metafield_definition, class_name: 'Spree::MetafieldDefinition'

  validates :product_type, :metafield_definition, presence: true
  validates :metafield_definition_id, uniqueness: { scope: :product_type_id }

  # Schema enforcement
  attribute :required, :boolean, default: false
  attribute :sort_order, :integer, default: 0

  scope :required, -> { where(required: true) }
  scope :ordered, -> { order(:sort_order) }
end
```

```ruby
# Renamed from Spree::OptionTypePrototype
class Spree::OptionTypeProductType < Spree.base_class
  belongs_to :option_type, class_name: 'Spree::OptionType'
  belongs_to :product_type, class_name: 'Spree::ProductType'

  validates :product_type, :option_type, presence: true
  validates :product_type_id, uniqueness: { scope: :option_type_id }

  # Live template: propagate to all products of this type
  after_create :add_option_type_to_products
  after_destroy :remove_option_type_from_products

  private

  # Add option type to all products of this type that don't already have it
  def add_option_type_to_products
    product_type.products.find_each do |product|
      product.option_types << option_type unless product.option_types.include?(option_type)
    end
  end

  # Remove option type from products — only if no variants use values from it
  def remove_option_type_from_products
    product_type.products.find_each do |product|
      has_variants_using = product.variants.joins(:option_values)
                                  .where(spree_option_values: { option_type_id: option_type.id })
                                  .exists?
      next if has_variants_using

      product.option_types.delete(option_type)
    end
  end
end
```

### Product changes

```ruby
class Spree::Product < Spree.base_class
  belongs_to :product_type, class_name: 'Spree::ProductType', optional: true

  # Remove: attr_accessor :prototype_id
  # On creation, sync option_types + categories from type
  after_create :sync_associations_from_product_type

  # New: validate required metafields on completion/activation
  validate :required_metafields_present, if: :active?

  # Prevent removing option types that come from the product type
  validate :type_option_types_present, if: :product_type

  # Check if an option type is inherited from the product type (cannot be removed)
  def type_option_type?(option_type)
    product_type&.option_types&.include?(option_type)
  end

  # Returns option types added beyond what the product type provides
  def extra_option_types
    return option_types unless product_type

    option_types - product_type.option_types
  end

  private

  def sync_associations_from_product_type
    return unless product_type

    # Add type's option types (additive — keeps any already assigned)
    product_type.option_types.each do |ot|
      option_types << ot unless option_types.include?(ot)
    end

    # Add type's categories (additive)
    product_type.categories.each do |cat|
      taxons << cat unless taxons.include?(cat)
    end
  end

  def type_option_types_present
    missing = product_type.option_types - option_types
    missing.each do |ot|
      errors.add(:option_types, "cannot remove type-defined option type: #{ot.name}")
    end
  end

  def required_metafields_present
    return unless product_type

    product_type.product_type_metafield_definitions.required.each do |ptmd|
      unless has_metafield?(ptmd.metafield_definition)
        errors.add(:base, "Missing required metafield: #{ptmd.metafield_definition.name}")
      end
    end
  end
end
```

### Admin UI (React SPA)

When editing a product with a ProductType:
1. Query ProductType's `option_types` and `metafield_definitions` (with `required` and `sort_order`)
2. Show type-defined option types as locked (cannot remove), extra option types as removable
3. Render metafield form fields in `sort_order`, grouped by namespace
4. Show required indicators on required fields
5. Validate required metafields before allowing product activation

When creating a product:
1. Optional ProductType selector
2. Selecting a type adds the type's option types and categories (live — not just a copy)
3. Shows the metafield form immediately

When editing a ProductType:
1. Adding an option type propagates to all products of that type
2. Removing an option type removes it from products (blocked if variants use values from it)
3. UI shows affected product count before confirming destructive changes

## Migration Path

### Phase 1: Rename (non-breaking)

Single migration:
```ruby
class RenamePrototypesToProductTypes < ActiveRecord::Migration[7.2]
  def change
    rename_table :spree_prototypes, :spree_product_types
    rename_table :spree_option_type_prototypes, :spree_option_type_product_types
    rename_table :spree_prototype_taxons, :spree_product_type_categories

    # Rename FK columns in join tables
    rename_column :spree_option_type_product_types, :prototype_id, :product_type_id
    rename_column :spree_product_type_categories, :prototype_id, :product_type_id
    rename_column :spree_product_type_categories, :taxon_id, :category_id

    # Add persistent FK on products
    add_column :spree_products, :product_type_id, :bigint
    add_index :spree_products, :product_type_id
  end
end
```

Model files:
- Rename `prototype.rb` → `product_type.rb`
- Rename `option_type_prototype.rb` → `option_type_product_type.rb`
- Rename `prototype_taxon.rb` → `product_type_category.rb`
- Add `Spree::Prototype = Spree::ProductType` alias in deprecation shim (remove in 6.1)
- Update Product: replace `attr_accessor :prototype_id` with `belongs_to :product_type`

## Constraints on Current Work

- **Do not remove Prototype model yet.** It will be renamed, not deleted. Don't add new features to it either.
- **Do not add `prototype_id` as a persisted column.** The rename migration will add `product_type_id` instead.
- **New MetafieldDefinitions should be resource_type-aware.** When creating definitions intended for products, use `resource_type: 'Spree::Product'` — this will matter when ProductType filters which definitions are available.
- **Don't create a separate Attribute/Property system.** Everything builds on existing Metafields. No new EAV tables.
- **Don't build new Product ↔ OptionType management that bypasses ProductType.** New admin UI for managing a product's option types should be type-aware: type-defined option types are locked, only extra option types can be freely added/removed.

## Open Questions

1. **Shared value pools for faceted filtering.** Currently metafield values are per-resource strings. For efficient faceted navigation (count products by "Color = Red"), we'd need shared value rows for choice-type metafields (e.g., dropdown/select types). This is a significant addition — should it be a future phase of this plan or a separate plan?

2. **Variant-level metafield definitions on ProductType.** Some platforms distinguish product-level vs variant-level attributes on the ProductType join. Should `ProductTypeMetafieldDefinition` carry a `level` column (`:product` or `:variant`)? OptionTypes already handle the variant dimension identity, so variant-level metafields would be descriptive metadata per variant (e.g., "variant weight note"). Defer or include?

3. **ProductType immutability enforcement.** Should changing `product_type_id` on a product be blocked at the model level, or just warned in the admin UI? We could allow it if no type-specific metafields have been set yet.

4. **Store API filtering by ProductType.** Should `GET /api/v3/store/products?filter[product_type_id]=pt_xxx` be a first-class filter? Useful for storefronts that want to show type-specific browse pages.

5. **Option type removal safety.** When removing an option type from a ProductType, the current design skips products whose variants use values from that option type. Should the admin API return a list of blocked products so the admin can resolve them, or should it hard-block the removal entirely until all variants are cleaned up?
