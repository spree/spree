## Summary

Add three new concepts to Spree's multi-store architecture:

- **Channel** â€” lightweight distribution surface within a Store (online, POS, Meta, wholesale portal). Controls where products are sold and how orders are attributed. Also owns the storefront **access posture** â€” whether anonymous visitors can see prices, browse at all, or check out as a guest.
- **Catalog** â€” product assortment + optional pricing override. Controls what's available and at what price for a specific audience (B2B tier, VIP group, regional assortment).
- **Company** â€” B2B customer hierarchy (Company â†’ CompanyLocation â†’ CompanyContact) for businesses with multiple buyers and offices.

`StoreProduct` is replaced by **ProductPublication** â€” the join between Product and Channel that says "this product is listed on this channel." The legacy `spree_products_stores` join table stays as compat surface for the [spree_multi_store](/developer/multi-store/quickstart) extension; **core no longer reads or writes it.**

`Product` is now `belongs_to :store` (single owner) in core. Multi-store catalog sharing â€” historically every `Product` belonged to many `Store`s via `spree_products_stores` â€” moves to the [spree_multi_store](/developer/multi-store/quickstart) extension. This matches Medusa/Saleor/Vendure: the catalog has one owner, distribution to multiple selling surfaces happens through channels.

## Problem

1. **No product visibility control within a store.** `StoreProduct` is binary â€” a product is in a store or it isn't. There's no way to say "this product is on the online store but not the POS" or "wholesalers see these 500 SKUs, retail sees 2000."

2. **No order attribution.** `Order.channel` exists as a free-text varchar but isn't connected to any model. You can't query "all POS orders this month" reliably or configure channel-specific behavior.

3. **No B2B company hierarchy.** `CustomerGroup` exists for user segmentation and pricing, but there's no way to model "Acme Corp has 3 offices, each with its own buyers, shipping addresses, and catalog assignments." B2B merchants need Company â†’ Location â†’ Contact.

4. **PriceList lacks a visibility counterpart.** PriceList controls pricing, but there's no matching concept for product visibility. A wholesaler needs both different prices AND a different product assortment.

## Current State

| What exists | What it does | Gap |
|---|---|---|
| `Store` | Config boundary â€” currency, tax, payments, fulfillment | Heavy, covers too much. No sub-store scoping. |
| `Market` | Geographic/currency region within a store | Read-only regions, no product visibility control |
| `StoreProduct` | Product â†” Store join (binary visibility) | No channel/catalog dimension |
| `CustomerGroup` | User segmentation within a store | No company hierarchy |
| `PriceList` + `PriceRules` | Conditional pricing (customer group, market, zone, volume, user) | No visibility counterpart |
| `Order.channel` | Free-text varchar | Not an FK, not a model |

## Architecture

Four concerns, four models:

```
Store (who you are â€” legal entity, config boundary)
  â†’ Channels (where you sell â€” online, POS, Meta, wholesale)
     â†’ ProductPublications (which products are listed on this channel)
  â†’ Catalogs (what you sell at what price â€” assortment + optional PriceList)
     â†’ CatalogProducts (which products are in this catalog)
     â†’ PriceList (optional pricing override â€” existing model)
  â†’ Markets (where your customers are â€” geography, currency â€” existing)
  â†’ Companies (B2B â€” who your business customers are)
     â†’ CompanyLocations (offices/branches with addresses)
     â†’ CompanyContacts (buyers â€” linked to user accounts)
```

**How they compose:**

- A **Channel** has a default Catalog (all listed products, base prices)
- A **Channel** can have additional Catalogs assigned to CustomerGroups or CompanyLocations
- A **Market** can have a default Catalog (regional assortment)
- A **CompanyLocation** can have assigned Catalogs (B2B per-location assortment + pricing)

**Product visibility resolution:**

```
1. Request arrives with: store, channel, user (optional)
2. Base set: ProductPublications for this channel
3. If user belongs to a CompanyLocation with Catalogs:
   â†’ Intersect with CatalogProducts from those Catalogs
4. Else if user belongs to a CustomerGroup with a Catalog:
   â†’ Intersect with CatalogProducts from that Catalog
5. Else: show all channel listings (default catalog)
```

**Price resolution (extends existing Pricing::Resolver):**

```
1. Check Catalog's PriceList (if Catalog has one)
2. Check other applicable PriceLists (existing PriceRule system)
3. Fall back to base price
```

## Key Decisions (do not deviate without discussion)

### Channel

- **Lightweight model** â€” not a config container. Store owns heavyweight config (currency, tax, payment methods, delivery methods). Channel is just a distribution surface.
- **Belongs to Store** (SingleStoreResource). One store can have many channels.
- **`Order.channel` becomes a FK** replacing the free-text varchar.
- **Each Channel can have its own PublishableApiKey** â€” so a storefront app, a POS app, and a Meta integration each get their own key scoped to their channel.
- **Default channel created per store** â€” "Online Store" is always present.
- **`storefront_access` (tiered enum) lives on Channel**, `nil` falling back to a store-level default of `public`. Gating is a per-channel posture, not store-wide â€” one store can run a public B2C channel and a `login_required` wholesale channel side by side. Enforced in the v3 Store API, not the storefront.
- **Channel gains its own `guest_checkout`**, `nil` â†’ store fallback; the `Store` preference is **kept** as the fallback (not removed â€” non-breaking), and the resolved value is enforced server-side in the v3 Store API for the first time.

### ProductPublication (replaces StoreProduct)

- **New `spree_product_publications` table.** Joins Product to Channel, not Product to Store.
- **No `store_id` column** â€” the publication's store is `channel.store` (delegated). Cross-store reporting joins through `spree_channels`.
- **Carries publication-specific data** â€” `published_at`, `unpublished_at` (scheduled publishing).
- **`position` was dropped from the original sketch** â€” nothing in 5.5 reads it and `acts_as_list` added a `MAX(position)` query per insert for no gain. Clean additive migration later if per-channel ordering ever becomes a real requirement.
- **Store-level scoping is direct** â€” `Spree::Store has_many :products` via `store_id` on Product. The historic `store.products through: :store_products` join is gone from core.

### Catalog

- **Belongs to Store.** A catalog can be shared across channels within the same store.
- **Has many CatalogProducts** (visibility) + optional **belongs_to PriceList** (pricing).
- **Catalog without PriceList** = assortment-only (use base prices). Catalog with PriceList = assortment + pricing.
- **Multiple Catalogs can apply** â€” when a CompanyLocation has 2 Catalogs, product visibility is the UNION of both. Pricing uses the first matching Catalog's PriceList (priority by position).
- **Default Catalog per Channel** â€” optional. If set, only products in the default catalog are visible on that channel. If not set, all ProductPublications are visible.

### Company (B2B)

- **Company â†’ CompanyLocation â†’ CompanyContact** hierarchy.
- **Company** belongs to Store. Represents a business customer (Acme Corp).
- **CompanyLocation** belongs to Company. Represents an office/branch. Has billing address, shipping address, tax exemption settings. Can have assigned Catalogs.
- **CompanyContact** belongs to CompanyLocation. Links to a User (Spree.user_class). A user can be a contact at multiple locations.
- **CustomerGroup is NOT replaced.** Companies and CustomerGroups serve different purposes: CustomerGroup is a flat segmentation tag (VIP, Wholesale). Company is a hierarchical B2B entity. A Company can be IN a CustomerGroup.

## Design Details

### Channel (shipped in 5.5)

See [`spree/core/app/models/spree/channel.rb`](../../spree/core/app/models/spree/channel.rb). Final shape:

- **`default` boolean column** + `scope :default` + partial unique index `(store_id) WHERE default = TRUE` on Postgres/SQLite (model-level uniqueness validation on MySQL). `before_validation :promote_first_channel_to_default` auto-promotes the first channel created on a store; `after_save :demote_other_defaults` enforces single-default invariant when an admin flips the flag.
- **No `default_catalog_id`** â€” waits for Catalog in 6.0.
- Includes routing-rules associations + `preference :order_routing_strategy` from `6.0-order-routing.md`.
- `#add_products(product_ids, published_at:, unpublished_at:)` bulk-UPSERTs publications. Idempotent: re-publishing an already-published product is a no-op for its window unless dates are explicitly passed (uses `Arel.sql('updated_at = <table>."updated_at"')` for the conflict clause to satisfy PG's DO UPDATE requirement without rewriting columns).
- `#remove_products(product_ids)` mirrors `add_products` â€” destroys publications, touches the channel.
- Touches every product via `Product.where(id: product_ids).touch_all` + `enqueue_search_index` after the upsert, since `upsert_all` bypasses `belongs_to :product, touch: true`.

### ProductPublication (shipped in 5.5 â€” new table)

See [`spree/core/app/models/spree/product_publication.rb`](../../spree/core/app/models/spree/product_publication.rb). Final shape:

- **New table `spree_product_publications`** with columns `product_id`, `channel_id`, `published_at`, `unpublished_at`, `created_at`, `updated_at`. No `store_id` column â€” `delegate :store, :store_id, to: :channel` for in-memory access; cross-store reporting queries join through `spree_channels`.
- **Unique index on `(product_id, channel_id)`**. `(product_id, channel_id, store_id)` was abandoned along with the in-place reshape.
- **`Spree::StoreProduct` deleted.** The class is gone in 5.5. Apps on 5.4 that referenced it directly must update to `Spree::ProductPublication`.
- **Legacy `spree_products_stores` table left untouched.** Core no longer reads or writes it. The upcoming [spree_multi_store](/developer/multi-store/quickstart) extension uses it to restore the `Product has_many :stores` association for merchants who need shared catalogs across stores.

`prefix :pp`, validates `(product, channel)`, validates `unpublished_at_after_published_at`, has a `published` scope.

### Product (shipped in 5.5 â€” single store ownership)

See [`spree/core/app/models/spree/product.rb`](../../spree/core/app/models/spree/product.rb) and [`spree/core/app/models/spree/product/channels.rb`](../../spree/core/app/models/spree/product/channels.rb). Final shape:

- **`belongs_to :store, optional: true`** with `before_validation :assign_default_store` (uses `Spree::Current.store` â†’ `Spree::Store.default` fallback).
- **`has_many :product_publications`** (autosave) + `has_many :channels, -> { distinct }, through: :product_publications`.
- **`accepts_nested_attributes_for :product_publications`** for Rails nested-attribute form writes (legacy admin Publishing card).
- **Custom `product_publications=`** setter for SPA's hash-array payload â€” full-set semantics (rows absent from the payload are destroyed). Coexists with the nested-attributes path on different keys.
- **Deprecated `available_on=` / `discontinue_on=`** setters write through to every publication's `published_at` / `unpublished_at`. Readers prefer the current-channel publication's value with a ternary fallback (`publication ? publication.send(attr) : super()`) â€” `||` would mask `nil` from an always-live publication.
- **`#stores` / `#stores=` / `#store_ids` / `#store_ids=` bridges** in `Spree::Product::LegacyMultiStoreSupport` (auto-included unless `SpreeMultiStore` defines), preserved as deprecation warnings for legacy callers.
- **`set_default_publication` callback removed.** New products created via the Admin API are *not* auto-published â€” the caller supplies `product_publications: [{ channel_id }]`. The dashboard SPA's create form does auto-publish on the store's default channel (Shopify-style); the legacy admin form has a Publishing card with the same UX.

### Product visibility query

```ruby
class Spree::Products::ForContext
  # Returns products visible to the current context
  def call(store:, channel:, user: nil)
    # Base: all active listings on this channel
    products = channel.products.active

    # If user has a company location with catalogs, filter to those
    if (location = user_company_location(user, store))
      catalog_ids = location.catalog_assignments.pluck(:catalog_id)
      if catalog_ids.any?
        catalog_product_ids = Spree::CatalogProduct.where(catalog_id: catalog_ids).select(:product_id)
        products = products.where(id: catalog_product_ids)
      end
    # If user belongs to a customer group with a catalog
    elsif (group_catalog_ids = user_group_catalog_ids(user, store)).any?
      catalog_product_ids = Spree::CatalogProduct.where(catalog_id: group_catalog_ids).select(:product_id)
      products = products.where(id: catalog_product_ids)
    end
    # Else: all channel listings (no catalog filtering)

    products
  end
end
```

### Gated Storefront Access (5.6)

Channel owns the storefront's **access posture** â€” the gate that Medusa, Saleor, and Vendure all push to the storefront layer. Spree enforces it in the **v3 Store API** instead, so it can't be bypassed by calling the API directly. Two orthogonal controls, both resolved from the channel the request is bound to (`X-Spree-Channel`, falling back to `store.default_channel`).

Because it builds only on the Channel infrastructure shipped in 5.5 (model, `preference` DSL, `X-Spree-Channel` resolution, v3 Store API auth context) and needs **no schema migration**, this ships in **5.6** â€” ahead of the Catalog/Company work it will later compose with in 6.0.

#### `storefront_access` â€” what anonymous visitors may see (tiered enum)

```ruby
# spree/core/app/models/spree/channel.rb
STOREFRONT_ACCESS = %w[public prices_hidden login_required].freeze

# Empty -> falls back to the Store-level preference (mirrors order_routing_strategy).
preference :storefront_access, :string, default: nil
validates :preferred_storefront_access, inclusion: { in: STOREFRONT_ACCESS }, allow_nil: true

# spree/core/app/models/spree/store.rb â€” the fallback default
preference :storefront_access, :string, default: 'public'
```

| Value | Products | Prices | Use case |
|---|---|---|---|
| `public` | visible to anyone | visible to anyone | today's behaviour, B2C |
| `prices_hidden` | browsable by anyone | **null for guests** | B2B2C teaser â€” "log in to see your price" |
| `login_required` | **401 for guests** | â€” | trade-only portal â€” anonymous sees nothing |

- Resolution lives directly in the `Spree::Channel::Gating` concern (there is no separate `HasStorefrontAccessPreference` concern): `channel.resolved_storefront_access` / `resolved_guest_checkout` return the channel value, or the store value when the channel preference is `nil`. `guest_checkout` is declared `nullable: true` so a blank write stays `nil` (inherit) rather than coercing to `false`.
- **"Guest" = no authenticated customer** â€” a request carrying only a publishable key (or a guest cart token) with no customer JWT. A logged-in customer is never gated.
- **Enforcement (v3 Store API):**
  - `prices_hidden` â†’ the gate injects a `hide_prices` serializer flag; `BaseSerializer#price_for`/`price_in` and the `money_attributes` helper (used by the product, cart, order, and line-item serializers) render every price/total field `null` for guest requests, so the cart can't leak what the catalog withholds. Logged-in customers (incl. B2B) serialize normally.
  - `login_required` â†’ a `before_action` on **both** `Store::BaseController` and `Store::ResourceController` (the two parallel Store API branches) returns `401` for guest requests across all Store endpoints. Public endpoints â€” sign in / register, password reset, and pre-login reference data (countries, currencies, locales, markets, newsletter, tokenized digital downloads) â€” opt out via `allow_guest_storefront_access!`.
- **This is the explicit "deny anonymous" gate the visibility model lacked.** In 5.6 it sits in front of channel-publication visibility (shipped 5.5): a `login_required` guest `401`s before any product list is built, and `prices_hidden` nulls prices on whatever the channel exposes. When Catalogs land in 6.0 the same gate fronts `Products::ForContext` â€” a guest never reaches the *"show all channel listings (default catalog)"* fallback, and a logged-in B2B buyer sees their company-location catalog at its PriceList price.

#### `guest_checkout` â€” may an order be placed without an account (channel override, store fallback)

```ruby
# spree/core/app/models/spree/channel.rb
# Empty -> falls back to the Store-level preference (same pattern as storefront_access).
preference :guest_checkout, :boolean, default: nil

# spree/core/app/models/spree/store.rb â€” retained as the fallback default (NOT removed)
preference :guest_checkout, :boolean, default: true
```

- **Channel overrides, Store falls back.** A B2C channel can allow guest checkout while a wholesale channel on the same store forbids it â€” the store is the wrong granularity once channels exist, but its preference is **kept as the fallback** so existing accessors (`store.prefer_guest_checkout`) and single-channel stores keep working unchanged. Resolution: `channel.resolved_guest_checkout` returns the channel value, or the store value when the channel preference is `nil`.
- **First server-side enforcement.** Today guest checkout is unenforced in the API â€” the legacy `Spree::Config.allow_guest_checkout` was *"only used in the rails frontend, not implemented in API"* and `Store#prefer_guest_checkout` only drove the legacy admin form. In 5.6 `Spree::Order#guest_checkout_disallowed?` (enforced in `Spree::Carts::Complete` so controller, payment-session, and webhook completion paths are all covered) requires an authenticated customer when the order's `channel.resolved_guest_checkout` is false.
- **`prices_hidden` implies no guest checkout.** `guest_checkout_disallowed?` also returns true whenever the channel hides prices from guests (`storefront_prices_hidden?`), regardless of the `guest_checkout` flag â€” a buyer who cannot see prices cannot meaningfully place an order. This dissolves the otherwise contradictory "prices hidden but guests may buy" configuration in the domain layer (no cross-record validation needed); the admin forms carry a hint saying guests must sign in to check out when prices are hidden.

#### Migration

- **No schema migration.** All three are model preferences stored in the existing serialized `preferences` column on `spree_channels` / `spree_stores` (same mechanism as `order_routing_strategy`).
- **No backfill required, non-breaking.** Channels default to `nil` â†’ store fallback for both `storefront_access` (â†’ `public`) and `guest_checkout` (â†’ the retained `Store` default), so existing behaviour is preserved until a merchant overrides a channel. The `Store#prefer_guest_checkout` accessor stays valid â€” it remains the fallback source.

### Gating Ă— Catalog composition

Channel **gating** (5.6) and the **Catalog/Company** layer (6.0) are orthogonal and run in sequence â€” the gate is the front door, the catalog is the personalized shelf behind it.

| Layer | Scope | Keyed on | Where it runs | Decides |
|---|---|---|---|---|
| Gate (`storefront_access` / `guest_checkout`) | whole Channel | authenticated vs guest | v3 Store API controller/serializer | can this request see the channel / see prices / check out |
| Catalog | per Company / CustomerGroup | membership | `Products::ForContext` + `Pricing::Resolver` | which products + which prices for this buyer |

**Ordering â€” the gate fires first.** `enforce_storefront_login_required!` is a `before_action`, so on a `login_required` channel a guest gets `401` *before* `Products::ForContext` runs â€” no catalog work for gated guests. `prices_hidden` sets `hide_prices` in `serializer_params`, which nulls whatever price the catalog resolver produced, so a company-catalog price is still withheld from a gated guest.

**Canonical B2B shape â€” one wholesale channel, many company catalogs:**
- Anonymous â†’ gate `401`s; the whole channel is sealed.
- Company A buyer (logged in) â†’ passes the gate â†’ `ForContext` intersects channel publications with Company A's location catalogs â†’ Catalog A assortment at Catalog A's PriceList.
- Company B buyer â†’ Catalog B assortment + pricing.
- Logged-in user with no company/catalog â†’ passes the gate â†’ channel default catalog (resolution step 5).

**`prices_hidden` + per-company PriceList = "log in to see your price."** A guest browses the default catalog with prices nulled; a logged-in company buyer sees prices, and the catalog layer makes them that company's negotiated prices. The two layers cooperate: the gate withholds from anonymous, the catalog personalizes per company.

**Key constraint â€” the gate is authenticated-vs-guest, not per-group.** `hide_prices?` / `login_required` only ask "is a customer logged in"; *any* authenticated customer passes. Consequences:
- The gate stops *anonymous* visitors, not *logged-in-but-unapproved* ones. There is no per-catalog or per-company `storefront_access`.
- Restricting what an unapproved-but-logged-in user sees is the **catalog's** job. For an "approved companies only" portal, give the channel an **empty/minimal default catalog** so step-5 fallback users see nothing purchasable â€” rather than expecting the gate to enforce approval.
- `guest_checkout` is channel-wide; fine for B2B since company buyers are always real users (`order.user.present?` holds).

**Design rule:** use **separate Channels for different access postures** (a public B2C channel + a `login_required` wholesale channel, each with its own publishable key); use **Catalogs for per-company variation within** a channel that shares one posture. Mirrors Shopify â€” B2B is a separate gated market, catalogs personalize assortment + pricing inside it.

## Migration Path

### Phase 1 â€” shipped in 5.5

**Migrations** (`spree/core/db/migrate/`):
- `20260508204040_create_spree_channels.rb` â€” Channel table (shipped 5.5 with order routing). Note: the in-migration `ensure_default_market` backfill that originally lived here was removed â€” default-channel seeding for existing stores moves to the `spree:channels:create_defaults` rake task.
- `20260601000001_create_spree_product_publications.rb` â€” **new** `spree_product_publications` table with `(product_id, channel_id, published_at, unpublished_at, created_at, updated_at)` + unique index on `(product_id, channel_id)`.
- `20260601000002_add_store_id_to_spree_products.rb` â€” adds `store_id` (nullable), `units_sold_count` (default 0), `revenue` (default 0) to `spree_products` + `(store_id, units_sold_count)` index. Existing rows have `store_id IS NULL` until the backfill rake runs.
- `20260602000001_add_default_to_spree_channels.rb` â€” adds `default` boolean to `spree_channels` + partial unique index on `(store_id) WHERE default = TRUE` (Postgres/SQLite; MySQL relies on the model-level uniqueness validation).

**The earlier additive-then-swap pattern (reshape `spree_products_stores` in place) was abandoned.** It carried a denormalized `store_id` column on the publication table and a `Spree::StoreProduct` STI alias for back-compat. The simpler shape â€” separate table, single owner on Product â€” is what actually shipped.

**Upgrade rake** (`spree:channels:upgrade` chains):
1. `spree:channels:create_defaults` â€” creates the default "Online Store" channel for every existing store (via `Store#ensure_default_channel`). Idempotent.
2. `spree:upgrade:populate_publications` â€” for every product with `store_id IS NULL`, reads the legacy `spree_products_stores` rows (raw SELECT with parameterized binding), picks a "home" store (preferring the row whose store is `default: true`, otherwise earliest by `created_at`), sets `spree_products.store_id`, and creates a `ProductPublication` row on each attached store's default channel. Runs in a transaction per product. Idempotent.
3. `spree:channels:backfill_order_channel_ids` â€” populates `spree_orders.channel_id` from the legacy `spree_orders.channel` string column.
4. `spree:channels:backfill_product_publication_dates` â€” copies the legacy `Product.available_on` / `Product.discontinue_on` columns (read raw via `product[:column]` to bypass the channel-aware reader) into each publication's `published_at` / `unpublished_at` where currently NULL.

**Deploy order**: code â†’ `db:migrate` â†’ `rake spree:channels:upgrade`. No swap migration â€” there's nothing to swap.

> **Warning**: Until `spree:channels:upgrade` runs, every product has `store_id IS NULL` and is invisible to `Product.for_store(store)`. The admin product list, storefront catalog, and search indexer all return empty. The upgrade guide [docs/developer/upgrades/5.4-to-5.5.mdx](../developer/upgrades/5.4-to-5.5.mdx) makes this prominent.

## Constraints on Current Work

- **`Spree::StoreProduct` is gone.** Use `Spree::ProductPublication` directly. Don't reintroduce the STI alias or any reference to `spree/core/app/models/spree/store_product.rb`.
- **`Spree::ProductPublication` has no `store_id` column.** It's delegated to `channel.store`. Don't write code that calls `ProductPublication.where(store_id: â€¦)` â€” use a join: `Publication.joins(:channel).where(spree_channels: { store_id: â€¦ })`.
- **`Product belongs_to :store` (single).** Don't write new code assuming `product.stores` is a real ActiveRecord association â€” the deprecation bridge returns `Array(store)` and won't chain. Multi-store catalogs live in [spree_multi_store](/developer/multi-store/quickstart).
- **`Channel#default` is a boolean column.** Use `Spree::Channel.default.where(store: â€¦)` or `store.default_channel`. Don't look up by `code: 'online'` â€” merchants can rename it.
- **Channel-aware scopes assume `Spree::Current.channel` is set.** API requests set it via the X-Spree-Channel header concern; jobs / rake tasks fall back to the legacy column path. Don't add new code that reads `Spree::Current.channel` without expecting either branch.
- **`Spree::Channel#add_products` is idempotent and preserves existing publication windows** when called without `published_at` / `unpublished_at` kwargs. Don't change this â€” bulk-publish ergonomics depend on it.
- **6.0 adds `default_catalog_id` to `spree_channels`** (additive â€” won't conflict with the 5.5 schema).
- **PriceList stays as-is.** Catalog optionally references a PriceList â€” don't duplicate pricing logic.
- **`storefront_access` is resolved per request from the bound channel** (`X-Spree-Channel` â†’ `store.default_channel`), with `nil` â†’ store fallback. Use `channel.resolved_storefront_access` for enforcement â€” don't read `channel.preferred_storefront_access` directly (it ignores the store fallback).
- **`guest_checkout` resolves channel-first, store-fallback.** The `Store` preference stays valid as the fallback (not removed); new enforcement code should resolve via `order.channel.resolved_guest_checkout` rather than reading `store.prefer_guest_checkout` directly.
- **Gating is enforced in the v3 Store API, not the storefront.** `prices_hidden` / `login_required` must hold for direct API calls â€” don't push the check into a storefront-only layer.
- **The gate is authenticated-vs-guest, not per-group.** Per-company assortment and pricing is the Catalog layer's job (`Products::ForContext`), which runs *after* the gate â€” don't add per-company/per-catalog logic to `storefront_access` or `hide_prices?`. See "Gating Ă— Catalog composition".

## Resolved Questions

1. **Default channel behavior.** Channel resolution via `X-Spree-Channel` request header (accepts channel `code`, e.g. `X-Spree-Channel: online`). If header is absent or doesn't match an active channel, falls back to `store.default_channel` (`Channel.default` scope on the `default` boolean column). Shipped in 5.5.

2. **How is the default channel identified?** Via a `default` boolean column on `Spree::Channel`, not by code. Partial unique index on Postgres/SQLite, model-level uniqueness validation on MySQL. The first channel created on a store auto-promotes; flipping the flag on a later channel auto-demotes the previous default. Looking up by `code: 'online'` would have been fragile (merchants can rename the default channel).

3. **Auto-publish on product create.** Admin API: **no** auto-publish â€” the caller supplies `product_publications: [{ channel_id }]`. Dashboard SPA's create form: **yes**, auto-publishes on `Channel#default == true` (Shopify-style). Sample data: explicitly publishes via `Loader#publish_sample_products`. The `set_default_publication` model callback was removed â€” moving the decision to the layer that knows the merchant's intent.

4. **Multi-store catalogs.** Core Spree 5.5+ models Product as `belongs_to :store` (single owner). Merchants who need shared catalogs across stores install the [spree_multi_store](/developer/multi-store/quickstart) extension, which restores `has_many :stores, through: :spree_products_stores` on top of the legacy join table. Matches the Medusa/Saleor/Vendure pattern (channels are the multi-brand primitive, stores are the tenant root).

5. **Cross-store onboarding via `Channel#add_products`.** Allowed: if a multi-store admin's API key has update permission on a product owned by sibling store A, calling `POST /admin/channels/:id/add_products` (where the channel belongs to store B) co-publishes the product onto store B's channel. The product's owning `store_id` doesn't change. This is the path by which the [spree_multi_store](/developer/multi-store/quickstart) extension's "publish across stores" UX threads through core.

6. **Catalog auto-sync.** Manual assignment only for 6.0, with bulk product assignment via Admin API (`POST /catalogs/:id/products/bulk`). Auto-sync using CollectionRule-style patterns is a future enhancement.

7. **CompanyContact roles and B2B approval workflows.** Out of scope for this plan. The 6.1 deliverable is the data model: Company, CompanyLocation, CompanyContact with a simple `role` string column. Role-based permissions, approval workflows, purchase limits, and invoice management are deferred to a dedicated B2B plan after 6.1.

8. **Gated storefront access model.** A single tiered `storefront_access` enum on Channel (`public` / `prices_hidden` / `login_required`), `nil` â†’ store fallback â€” rather than independent booleans (which permit redundant/contradictory combinations) or a prices-only boolean (which can't express a full login wall). Enforced in the v3 Store API: `prices_hidden` nulls price fields for guests, `login_required` `401`s catalog reads for guests. This is the explicit "deny anonymous" gate the visibility resolver lacked. Targeted for **5.6** â€” it builds only on the 5.5 channel infrastructure and needs no schema change, so it ships ahead of the 6.0 Catalog/Company work.

9. **Where guest checkout lives.** Channel gains a `guest_checkout` preference (`nil` â†’ store fallback), so B2C and wholesale channels on one store can differ; the `Spree::Store` preference is **retained as the fallback** for old accessors (not removed â€” non-breaking). The v3 API enforces the resolved value server-side for the first time (it was previously a legacy-frontend-only preference).
