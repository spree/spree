# Analytics Semantic Layer

**Status:** Draft
**Target:** Spree 6.0 (Phase 1–2) → 6.1 (Phase 3 facts adapter GA, Phase 4 AI) — Enterprise ClickHouse adapter out of band
**Depends on:** `6.0-admin-api.md` (v3 conventions), `5.5-admin-api-key-scopes.md` (scope gating), `6.0-multi-vendor-marketplace.md` (vendor forced scope + vendor dashboard consumer), `5.5-admin-api-cli.md` (first agent surface), `5.6-solid-queue-single-container.md` (fact writes + reconciliation jobs)
**Author:** Damian + Claude
**Last updated:** 2026-07-22

## Summary

Replace per-feature, hand-rolled analytics — the legacy admin's `AnalyticsConcern`, the v3 dashboard serializers, and the `Spree::Report`/`ReportLineItem` STI authoring model — with one semantic layer: a **registry of metrics and dimensions** (the developer extension point), a **query contract + compiler** (one `POST /api/v3/admin/analytics/query` endpoint; a saved report is a stored query + viz config), and **pluggable storage adapters** (live OLTP by default — zero new infra; incremental fact tables opt-in; ClickHouse as an Enterprise adapter). Developers extend the *vocabulary*; merchants, the dashboard, saved reports, vendor dashboards, and the AI layer all compose *sentences* from it. Store, currency, and vendor scoping are enforced by the compiler below every consumer — including the LLM.

## Problem

Spree currently has **three parallel analytics implementations**, none extensible at the right granularity:

| Surface | Implementation | Problems |
|---|---|---|
| Legacy admin home | `Spree::Admin::AnalyticsConcern` — inline AR aggregates | Maintenance mode; duplicates definitions (its `calc_growth_rate` disagrees with the v3 serializer's `growth_rate` on the zero-baseline case) |
| React dashboard home | `Spree::Api::V3::Admin::Dashboard{Analytics,Rankings,Operations}Serializer` — hand-rolled SQL POROs | Every new number touches serializer + SDK + UI + locales; "revenue," channel scoping, and period comparison were each defined ad hoc and had to be deduplicated by review; chart buckets in UTC while `Spree::Report` uses store timezone |
| Reports | `Spree::Report` STI subclasses + `ReportLineItem` | Developers must write a whole report class per question; output is row-level CSV, not aggregates; no composition, no viz, no API-driven querying |

The 5.6 dashboard work (branch `claude/react-dashboard-home-reporting-71weaz`) is the proof: shipping a Pulse-style home screen required inventing revenue semantics (`SUM(pre_tax_amount)` vs `quantity * price`), a nil-tolerant channel filter (`Order.for_channel`), a previous-period alignment convention, and a growth-rate convention — all as one-off code. The upcoming multi-vendor marketplace (6.0 headline) needs vendor dashboards that would otherwise be a *fourth* copy.

## Key Decisions (do not deviate without discussion)

### 1. Developers extend vocabulary, not reports

The extension point is a registry of metrics and dimensions, mirroring `Spree.order_routing.rules` / `Spree::Dependencies`:

```ruby
Spree.analytics.metric :net_revenue,
  sql: "SUM(#{Spree::LineItem.table_name}.pre_tax_amount)",
  base: :line_items,
  format: :money                       # money metrics REQUIRE the currency forced scope

Spree.analytics.metric :orders_count, sql: 'COUNT(DISTINCT spree_orders.id)', base: :orders, format: :integer

Spree.analytics.dimension :channel, base: :orders, column: :channel_id, lookup: 'Spree::Channel'
Spree.analytics.dimension :category,
  base: :line_items, joins: [variant: { product: :classifications }],
  column: 'spree_products_taxons.taxon_id', lookup: 'Spree::Category'
Spree.analytics.dimension :completed_at, base: :orders, type: :time, grains: %i[day week month quarter]
```

A definition carries: SQL fragment (metrics), join graph, type/format, allowed grains, lookup model for label/prefixed-id resolution, and authorization metadata. Registered definitions are the **allowlist** — the only things a query may reference.

### 2. Query contract stolen from Cube.dev/Lightdash, not invented

```json
{ "metrics": ["net_revenue", "orders_count"],
  "dimensions": [{ "name": "completed_at", "grain": "week" }, "channel"],
  "filters": [{ "dimension": "channel", "op": "eq", "value": "ch_x7Yq2k" }],
  "time_range": { "since": "-90d" },
  "compare": "previous_period",
  "sort": "-net_revenue",
  "limit": 50 }
```

One endpoint: `POST /api/v3/admin/analytics/query`, gated by a `read_analytics` secret-key scope or CanCanCan for admin JWTs. A **saved report** is this JSON + viz config as a store-scoped DB row (`Spree::Analytics::SavedReport`) — shareable, versionable, seedable. The dashboard home becomes seeded saved reports.

### 3. Forced scopes: store, currency, vendor — enforced by the compiler

- **Store**: every compiled query is rooted in `current_store` associations (`store.orders`, `store.line_items`, …). Non-negotiable, never expressible or removable via the contract.
- **Currency**: any query touching a `format: :money` metric is compiled with exactly one currency filter (param or store default). Cross-currency aggregation is a **422**, not a silently wrong sum.
- **Vendor**: when the caller is vendor-scoped (`6.0-multi-vendor-marketplace.md`), the compiler injects the `vendor_id` filter the way `Spree::Reports::SalesTotal` does by hand today. Vendor dashboards are the same saved reports running under a forced scope — not a second report set.

Enforcement lives **below** every consumer, including the AI layer. Never by prompt, never per-serializer.

### 4. Validation rejects; it never silently narrows

Unknown metric/dimension/grain/op → 422 with the v3 error shape naming the offender. This is the lesson from the silently-dropped Ransack predicates bug (fixed 2026-07-22): a filter that doesn't apply must fail loudly, because a query that "works" while ignoring a filter is worse than an error.

### 5. Storage is a pluggable adapter; live OLTP is the default

`Spree::Analytics::Adapters::Live` compiles to Arel against the transactional schema — zero new infra, correct on day one for the vast majority of installs (the current dashboard proves viability: the whole home payload is ~a dozen aggregate queries). `Adapters::Facts` (incremental rollup tables) is **opt-in** and Phase 3. ClickHouse is an Enterprise adapter behind the same interface, never a core dependency. **Live-first sequencing is deliberate** (see decisions.md 2026-07-22): the contract is identical across adapters, so storage can be swapped without touching consumers, and the extension point ships a release earlier.

### 6. Grain SQL belongs to the adapter, not the definition

Spree supports PostgreSQL, MySQL, and SQLite. `date_trunc` vs `strftime` vs `DATE_FORMAT` is adapter code. Metric SQL fragments must stay portable (plain aggregates over portable expressions); time bucketing is compiler/adapter-owned.

### 7. Time grains resolve in the store's timezone

`Spree::Report` already does this (`store.preferred_timezone`); the 5.6 dashboard serializers bucket in UTC. The semantic layer standardizes on store timezone for day+ grains and fixes the dashboard inconsistency when it migrates.

### 8. Money is decimal end-to-end; formatting at the edge

Aggregates compute on decimal columns (`pre_tax_amount`, `total`). Floats never touch storage or the wire until final display values (`Spree::Money` / client formatting). This also rules out `ankane/rollup`'s float value column as a fact store (see References).

### 9. AI layer: NL → query JSON, never NL → SQL

The registry serializes into the tool schema (same source as the OpenAPI spec), so the model cannot reference a metric that doesn't exist; validation errors are recoverable in the agent loop; forced scopes hold below the LLM. Read-only — no mutation confirmation gate needed. First agent surface is the CLI (`spree api post /analytics/query`, per `5.5-admin-api-cli.md`), ahead of any in-app agent or MCP.

### 10. `Spree::Report` stays; its authoring role ends

`Spree::Report` is a row-level CSV exporter with attachment/mailer plumbing — orthogonal, keeps working. It is no longer the thing developers extend to answer aggregate questions; new report *types* for analytics purposes are not accepted into core once Phase 1 ships. (Long term the CSV export path can consume the semantic layer for aggregate exports; row-level exports remain `Spree::Report`'s job.)

### 11. No Blazer or rollup dependency

`ankane/blazer` is analyst-facing raw SQL (no semantic guardrails, no per-store row scoping, engine UI not embeddable in the SPA) — the opposite trust model; it may be *documented* as a self-hosted ops option. `ankane/rollup` is disqualified as a backbone (single float value column, one metric per series name, jsonb dimensions are Postgres-only) but its recompute-current-period write model is the reference for the facts adapter. We steal designs (Blazer's checks/alerts, saved-query rows; rollup's incremental semantics), not dependencies.

## Design Details

### Registry (`spree/core`)

- `Spree::Analytics::Registry` with `metric(name, **opts)` / `dimension(name, **opts)`, exposed as `Spree.analytics`. Definitions are frozen structs; re-registration under the same name raises outside of an explicit `replace:` flag.
- **Bases** name the root relation the compiler starts from, always store-scoped: `:orders` → `store.orders.complete`, `:line_items` → `store.line_items` (joined through completed orders). A query's members must share a compatible base; the compiler picks the finest.
- **Starter vocabulary (core-registered):** metrics `net_revenue` (pre_tax_amount), `gross_revenue` (order totals), `orders_count`, `units_sold`, `customers_count` (distinct email), `aov` (derived: `gross_revenue / orders_count`); dimensions `completed_at` (time), `channel`, `category`, `product`, `market`, `country`, `payment_status`, `fulfillment_status`; marketplace registers `vendor` when installed.
- **Derived metrics** reference other metrics (`aov`), computed post-aggregation so division-by-zero and rounding conventions live in one place (nil-growth convention from the 5.6 dashboard carries over: no baseline → `null`, client renders "New").

### Compiler + endpoint (`spree/api`)

- `Spree::Analytics::Query` — validates the JSON against the registry (unknown member/op/grain → 422 listing valid options), normalizes time ranges (`-90d`, ISO dates), resolves prefixed-id filter values via each dimension's lookup, applies forced scopes, and hands a normalized plan to the adapter.
- `compare: "previous_period"` compiles both periods and returns aligned series + per-metric growth — replacing the hand-rolled `previous_time_range`/day-offset logic in `DashboardAnalyticsSerializer`.
- Response rows carry raw values plus display values for `:money` metrics; dimension values carry prefixed id + resolved label (`{ "id": "ch_x", "label": "Wholesale" }`).
- Result caching: `Rails.cache` keyed on (store, normalized query hash, adapter), short TTL; per-query `cache: false` escape hatch.
- Guardrails: `limit` capped (≤ 1000), max dimensions per query, statement timeout on the live adapter.

### Saved reports (`spree/core` + `spree/api`)

- `Spree::Analytics::SavedReport < Spree.base_class`: `store_id`, optional `user_id`, `name`, `query` (JSON), `viz` (JSON: chart kind, metric→axis mapping), `seeded` flag; `Spree::SingleStoreResource`; prefixed id (`sq_`); lifecycle events. CRUD at `/api/v3/admin/analytics/saved_reports`.
- The dashboard home's KPI/chart/rankings cards are seeded rows; drill-down links derive from the saved query's filters (removing the count-vs-deep-link duplication the 5.6 operations card has).

### Storage adapters

```ruby
class Spree::Analytics::Adapters::Base
  # plan: normalized query (members resolved to definitions, scopes applied)
  # returns rows of { dimension_values..., metric_values... }
  def execute(plan) = raise NotImplementedError
end
```

- **`Live`** (default): Arel against OLTP through store associations. Ships Phase 1.
- **`Facts`** (opt-in, Phase 3): typed daily-grain tables — `spree_analytics_order_facts` (store_id, channel_id, vendor_id, market_id, currency, day, decimal metric columns, counts) and `spree_analytics_line_item_facts` (adds product_id/taxon_id). Written incrementally from lifecycle events (`order.completed`, `order.canceled`, refund events — subscribers in `app/subscribers/spree/`), recompute-the-touched-day semantics (rollup-style), **refunds/cancellations restate the original day** (Pulse's accounting rule), nightly reconciliation + backfill via rake tasks (never migrations). Falls back to Live for members without fact coverage.
- **Enterprise ClickHouse**: same interface, out of scope for this repo beyond keeping the seam honest.
- Selection via `Spree::Dependencies.analytics_adapter`, per-store override possible later.

### Dashboard + SDK (`packages/*`)

- `client.analytics.query(json)` + `client.analytics.savedReports.*` in `@spree/admin-sdk`; hand-written types for the contract (no Alba serializer → no typelizer).
- `packages/dashboard`: a `useAnalyticsQuery(query)` hook; chart components take contract result shape directly; a Reports SPA page (nav slot `admin.nav.reports` already exists, unused) lists saved reports and renders them; "edit query" UI can start as JSON-lite (metric/dimension/filter pickers driven by a registry-introspection endpoint: `GET /analytics/schema`).
- The three `dashboard/*` endpoints from 5.6 stay as thin compat shims during migration, then the home page moves to saved reports and the shims are dropped in 6.0 final.

### AI layer (Phase 4)

- `GET /analytics/schema` (registry introspection) doubles as the tool-schema source. The agent tool is literally "run_analytics_query(json)" + "list/save report". "Why did EU revenue dip last week?" = a few compiled queries + a rendered chart + an offer to save. Ships against the CLI first, in-app agent second.

## Migration Path

1. **Phase 1 (6.0):** Registry + `Query` compiler + `Live` adapter + `POST /analytics/query` + `GET /analytics/schema` + `read_analytics` scope. Core starter vocabulary. Controller specs are the contract's test suite.
2. **Phase 2 (6.0):** `SavedReport` model + CRUD + seeds; SDK + `useAnalyticsQuery`; Reports SPA page; reimplement dashboard home on saved queries (keep `dashboard/*` endpoints as shims → delete before 6.0 final). Marketplace registers the `vendor` dimension + forced scope; vendor dashboard = same seeded reports.
3. **Phase 3 (6.1):** `Facts` adapter behind the same contract: schema, event subscribers, reconciliation/backfill rakes, restatement rules. Opt-in flag; Live remains default.
4. **Phase 4 (6.1):** AI layer (tool schema from registry, agent loop, CLI-first). Blazer-style scheduled checks/alerts on saved reports as a follow-up plan.

Legacy surfaces: `Spree::Admin::AnalyticsConcern` untouched (legacy admin is maintenance-mode); `Spree::Report` untouched per Decision 10.

## Constraints on Current Work

- **No new hand-rolled aggregate endpoints.** New analytics needs go through this plan; the 5.6 `dashboard/*` serializers may be extended only for home-screen fixes, not new report surfaces.
- **Revenue means `SUM(pre_tax_amount)`** (`DashboardSerializerHelpers#revenue_sum_sql`) — keep it single-sourced; it becomes the `net_revenue` metric.
- **Keep forced-scope discipline**: all analytics queries go through store associations (`store.orders`, `store.line_items`, …), filter exactly one currency, and use `Order.for_channel` for optional channel scoping.
- **Don't add new `Spree::Report` STI subclasses for aggregate questions**; row-level CSV exports only.
- **New time-bucketed queries must document their timezone**; prefer store timezone (Decision 7).
- **Don't adopt Blazer/rollup as dependencies** (Decision 11).

## Open Questions

- **Fact currency handling**: per-currency fact rows (query one currency at a time, mirroring Live) vs converted-at-write "reporting currency" columns. Leaning per-currency rows; conversion is a display concern until merchants demand blended totals.
- **Store timezone changes** restate historical day buckets in the facts adapter — accept + document, or version the timezone per row?
- **SavedReport permissions**: store-wide vs per-user visibility; how vendor-scoped saved reports appear in the vendor dashboard.
- **Checks/alerts** (Blazer-style scheduled saved-report thresholds → email/webhook): follow-up plan or Phase 3 rider?
- **Registry naming**: `Spree.analytics` vs `Spree.reporting`; `SavedReport` vs collision-adjacent `Spree::Report` (prefix `sq_` vs `rep_`).
- **Operations-card counts** (orders to fulfill, low stock) are point-in-time state, not time-series analytics — do they stay a bespoke endpoint or become "state metrics" (no time dimension) in the registry?
