<Info>
The dashboard plugin API ships in **Spree 5.5** as part of the new Vite-based admin dashboard (currently developer preview). 
For the stable admin dashboard refer to the [extension model](/developer/admin/extending-ui).
</Info>

A dashboard plugin is an npm package that extends the Spree admin without forking it. Plugins can:

- Add **sidebar entries** that link to plugin-owned pages
- Mount **custom routes** under `/$storeId/...`
- Inject **slot widgets** into core pages (e.g. a card on the product detail page)
- Add **columns to existing tables** (Products, Orders, Customers, …)
- Register **their own resource tables** for use with `<ResourceTable>`
- Contribute **translations** that merge into the framework's i18next namespace

A working reference implementation lives in [`packages/dashboard-plugin-example/`](https://github.com/spree/spree/tree/main/packages/dashboard-plugin-example) in the Spree monorepo — a Brands plugin that exercises every extension point. The Spree CLI scaffolds the same shape for any new plugin.

## Quick start: scaffold with `@spree/cli`

```bash
npx @spree/cli plugin new brands
```

The CLI walks you through the prompts (plugin name, npm scope, author, license) and produces a monorepo with the dashboard plugin pre-wired. It picks the npm package name, peer dependencies, and `defineDashboardPlugin` skeleton based on your answers.

```bash
brands/
├── package.json                     # workspace root (e.g. "@acme/brands")
├── pnpm-workspace.yaml
├── README.md
├── LICENSE
└── packages/
    └── dashboard/                   # the dashboard plugin (e.g. "@acme/brands-dashboard")
        ├── package.json
        ├── tsconfig.json
        ├── biome.json
        └── src/
            ├── index.tsx            # entry — defineDashboardPlugin call
            ├── types.ts             # TypeScript shapes for your records
            ├── client.ts            # adminClient.request wrappers
            ├── routes/
            │   └── brands-list.tsx  # custom route mounted at /$storeId/brands
            ├── slots/
            │   └── product-brands-card.tsx
            └── locales/
                └── en.json
```

Flags:
- `--no-dashboard` — skip the dashboard plugin half
- `--no-install` — don't run `pnpm install` after scaffolding
- `--force` — overwrite a non-empty destination directory

The CLI also reserves `--no-engine` for the upcoming Rails-engine scaffolding (the Ruby half that ships your Admin API endpoints). For now, generate the Ruby side with the existing [`spree-extension`](https://github.com/spree/spree_extension) gem alongside the CLI's dashboard output.

## Anatomy of a plugin

If you're not using the CLI, the minimum layout is:

```
my-spree-plugin/
├── package.json                     # peer deps + version
├── tsconfig.json
└── src/
    ├── index.tsx                    # entry — defineDashboardPlugin call
    ├── types.ts                     # TypeScript shapes for your records
    ├── client.ts                    # adminClient.request wrappers
    ├── routes/
    │   └── my-list-page.tsx         # custom route(s)
    ├── slots/
    │   └── my-widget.tsx            # slot widget(s)
    └── locales/
        └── en.json                  # translations
```

### package.json — peer dependencies

```jsonc
{
  "name": "spree-brands-plugin",
  "version": "0.1.0",
  "type": "module",
  "main": "./src/index.tsx",
  "exports": { ".": "./src/index.tsx" },
  "peerDependencies": {
    "@spree/admin-sdk": "^0.1.0",
    "@spree/dashboard-core": "^0.1.0",
    "@spree/dashboard-ui": "^0.1.0",
    "@tanstack/react-query": "^5",
    "@tanstack/react-router": "^1",
    "i18next": "^26",
    "react": "^19",
    "react-i18next": "^17"
  }
}
```

**Always declare these as peer deps, not regular deps.** The dashboard's four registries (nav, route, slot, table) are module singletons — if your plugin bundles its own copy of `@spree/dashboard-core`, your `nav.add()` call writes to a different Map than the dashboard reads. pnpm/yarn dedupe peer deps to a single instance and that's the only setup that works.

### src/index.tsx — the entry

The plugin's entry calls `defineDashboardPlugin` once at module load. The dashboard imports the plugin as a side-effect:

```tsx
// my-spree-plugin/src/index.tsx
import { defineDashboardPlugin, i18n } from '@spree/dashboard-core'
import { PackageIcon } from 'lucide-react'
import en from './locales/en.json'
import { BrandsListPage } from './routes/brands-list'
import { ProductBrandCard } from './slots/product-brand-card'

// 1. Merge translations into the dashboard's existing `admin.*` namespace.
i18n.addResourceBundle('en', 'translation', en, true, true)

// 2. Register everything else.
defineDashboardPlugin({
  nav: [{
    key: 'brands',
    label: i18n.t('admin.brands_plugin.nav'),
    path: '/brands',
    icon: PackageIcon,
    position: 35,
    subject: 'Spree::Brand',  // hide from admins without read permission
  }],
  routes: [{
    key: 'brands',
    path: '/brands',
    component: BrandsListPage,
    subject: 'Spree::Brand',
  }],
  slots: {
    'product.form_sidebar': [
      { id: 'brand-card', component: ProductBrandCard, position: 250 },
    ],
  },
  tables: {
    products: { add: [
      { key: 'brand', label: 'Brand', render: (p) => p.brand_id ?? '—' },
    ]},
  },
})
```

The app activates the plugin with a single side-effect import in `main.tsx`, **before the router mounts**:

```tsx
// packages/dashboard/src/main.tsx
import 'spree-brands-plugin'  // registers everything above
// ...rest of the dashboard bootstrap
```

## Extension points in detail

### 1. Navigation (`nav`)

Add a sidebar entry. The dashboard prefixes `path` with `/$storeId` automatically:

```ts
nav: [{
  key: 'brands',
  label: 'Brands',
  path: '/brands',
  icon: PackageIcon,
  position: 35,        // 100/200/300 = built-ins; pick a number between
  subject: 'Spree::Brand',  // CanCanCan subject; omit for always-visible
}]
```

The `subject` field hides the entry for admins who can't `read` the subject. Server-side enforcement still applies — this is UX, not authorization.

`children` lets you nest entries under a top-level parent:

```ts
{ key: 'brands', label: 'Brands', children: [
  { key: 'all-brands', label: 'All brands', path: '/brands' },
  { key: 'featured-brands', label: 'Featured', path: '/brands?featured=1' },
]}
```

### 2. Routes (`routes`)

Mount a page under `/$storeId/...`. The dashboard's catch-all dispatcher reads from the route registry on every navigation:

```tsx
routes: [{
  key: 'brands',
  path: '/brands',
  component: BrandsListPage,
  subject: 'Spree::Brand',  // 403 fallback when permission denied
}]
```

Path patterns support TanStack-Router-style params:

```ts
{ key: 'brand-detail', path: '/brands/$brandId', component: BrandDetailPage }
```

Your component receives `{ params, storeId, searchParams }`:

```tsx
function BrandDetailPage({ params, storeId }: {
  params: Record<string, string>
  storeId: string
  searchParams: Record<string, unknown>
}) {
  const brandId = params.brandId
  // ...
}
```

### 3. Slots (`slots`)

Slots are named injection points the dashboard exposes on built-in pages. The full catalogue lives in `@spree/dashboard`'s source — common slots include:

- `product.form_sidebar` — right sidebar on the product detail page
- `product.actions` — primary action buttons on the product page
- `order.page_body` — main column of the order detail page
- `customer.tabs` — additional tabs on the customer detail page

Register a component for a slot:

```tsx
slots: {
  'product.form_sidebar': [{
    id: 'brand-card',
    component: ProductBrandCard,
    position: 250,
    if: (ctx) => ctx.permissions?.can('read', 'Spree::Brand'),
  }],
}
```

The slot's context (`product`, `permissions`, `store`, `user`, …) is passed to the component as props. Each slot documents its context shape.

### 4. Tables — extend existing (`tables.<key>.add`)

Add columns to the dashboard's built-in resource tables:

```ts
tables: {
  products: {
    add: [{
      key: 'brand',
      label: 'Brand',
      sortable: false,
      filterable: false,
      default: false,  // off by default; admin can toggle on
      render: (product) => product.brand_id ? <BrandName id={product.brand_id} /> : '—',
    }],
    // Or remove/update existing columns:
    remove: ['legacy_field'],
    update: { total_price: { label: 'Net total' } },
  },
}
```

If the target table isn't registered yet (route hasn't loaded), the mutation is queued and replayed when `defineTable('products', ...)` runs. You don't need to know the order.

### 4b. Tables — register your own

For plugin-owned tables (used by your custom routes), use `defineTable`:

```tsx
import { defineTable } from '@spree/dashboard-core'

defineTable<Brand>('brands', {
  title: 'Brands',
  searchParam: 'search',
  searchPlaceholder: 'Search by name…',
  defaultSort: { field: 'name', direction: 'asc' },
  columns: [
    { key: 'name', label: 'Name', sortable: true, default: true,
      render: (b) => <Link to={`/$storeId/brands/${b.id}`}>{b.name}</Link> },
    { key: 'products_count', label: 'Products', default: true,
      render: (b) => b.products_count },
  ],
})
```

Then mount `<ResourceTable tableKey="brands">` from your route page. See [`packages/dashboard-plugin-example/src/routes/brands-list.tsx`](https://github.com/spree/spree/tree/main/packages/dashboard-plugin-example/src/routes/brands-list.tsx) for a runnable example.

### 5. Translations

Merge your keys into the framework's namespace via `i18n.addResourceBundle`:

```tsx
import { i18n } from '@spree/dashboard-core'
import en from './locales/en.json'

i18n.addResourceBundle('en', 'translation', en, true, true)
```

```jsonc
// src/locales/en.json
{
  "admin": {
    "brands_plugin": {            // namespace under admin.<your-plugin>.*
      "nav": "Brands",
      "page": { "title": "Brands", "subtitle": "Manage product brands" }
    }
  }
}
```

**Always put plugin text under your own namespace** (`admin.<your-plugin>.*`). Writing to `admin.actions.save` would overwrite the framework's translation — two plugins doing that would race. The dashboard's base namespace (`admin.actions.*`, `admin.common.*`, `admin.fields.<simple>.*`, `admin.validation.*`) is available for you to read but should not be extended.

## Talking to a backend

Your plugin needs a backend to talk to. The dashboard's pre-built SDK methods (`adminClient.products.list()`, `adminClient.orders.get(id)`, etc.) only cover Spree's first-party resources. For your own endpoints, use the SDK's `request` escape hatch:

```ts
// my-spree-plugin/src/client.ts
import { adminClient } from '@spree/dashboard-core'
import type { PaginatedResponse } from '@spree/admin-sdk'
import type { Brand } from './types'

export const brandsClient = {
  list: (params?: Record<string, unknown>) =>
    adminClient.request<PaginatedResponse<Brand>>('GET', '/brands', { params }),
  get: (id: string) =>
    adminClient.request<Brand>('GET', `/brands/${id}`),
  create: (body: { name: string; slug?: string }) =>
    adminClient.request<Brand>('POST', '/brands', { body }),
}
```

Paths are relative to `/api/v3/admin`. The request method uses the same auth (secret key or JWT), retry logic, and 401-refresh handshake as the built-in resources.

The backend half is a normal Spree extension gem: a model, a Rails controller subclassing `Spree::Api::V3::Admin::ResourceController`, an Alba serializer. See [Custom Admin Endpoints](/developer/sdk/admin/extending) and the [Admin API customization](/developer/admin/admin) guide.

## Order-of-operations

Three things must happen at boot, in this order:

1. **i18n bootstrap** — `@spree/dashboard-core`'s `i18n.ts` runs (via the dashboard's `i18n-setup.ts` side-effect import) and registers the base translation namespace.
2. **Plugin imports** — each `import 'my-plugin'` runs `defineDashboardPlugin`, populating registries.
3. **Router mounts** — the dashboard's `RouterProvider` reads the route registry on first navigation.

If a plugin runs before i18n, `i18n.t(...)` returns the literal key string instead of a translation. If a plugin runs after the router mounts, route registration still works (the catch-all reads on every navigation), but nav entries appear "for new visitors only" rather than from boot. **Always import plugins right after `@/i18n-setup` in `main.tsx`.**

## Permission gating

Three places to declare permissions:

- **Nav entry** `subject`: hides the sidebar entry
- **Route entry** `subject`: 403 fallback at the route
- **Slot entry** `if`: visibility predicate, receives the merged context

```ts
nav: [{ key: 'brands', subject: 'Spree::Brand', /* ... */ }],
routes: [{ key: 'brands', subject: 'Spree::Brand', /* ... */ }],
slots: {
  'product.form_sidebar': [{
    id: 'brand-card',
    component: ProductBrandCard,
    if: (ctx) => ctx.permissions?.can('read', 'Spree::Brand'),
  }],
}
```

All three are UX-only. The backend's CanCanCan rules still gate the actual API responses — the plugin should treat permission checks as a hint about what to render, not a security boundary.

## Distributing

Publish to npm under your own scope (e.g. `@my-org/spree-brands-plugin`). Document the install:

```bash
pnpm add @my-org/spree-brands-plugin
```

Plus the side-effect import in the dashboard's `main.tsx`:

```tsx
import '@my-org/spree-brands-plugin'
```

Pair the npm package with your Rails-side extension gem so consumers install both. Once both are in place, every store running your customers' dashboards picks up your feature.
