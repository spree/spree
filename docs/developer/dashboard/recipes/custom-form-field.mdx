The dashboard's product form lives in `@spree/dashboard` and renders core attributes (name, description, price, …). When you've added a custom column to `spree_products` or a custom field through a [metafield definition](/developer/customization/metadata), you'll want it to surface in the same form. This recipe walks the four pieces end-to-end.

## Backend prerequisites

This recipe assumes you've added the column or metafield on the Rails side:

```ruby
# A migration:
class AddTechSpecsToSpreeProducts < ActiveRecord::Migration[7.2]
  def change
    add_column :spree_products, :tech_specs, :text, null: false, default: ''
  end
end
```

And declared it as a permitted attribute on the Admin API serializer + controller (see [Admin API conventions](/developer/dashboard/customization/backend)).

## 1. Register the field

Use a slot to inject your field into the product form's sidebar:

```tsx
// packages/dashboard/src/customizations.ts
import { defineDashboardPlugin } from '@spree/dashboard-core'
import { TechSpecsField } from './fields/tech-specs-field'

defineDashboardPlugin({
  slots: {
    'product.form_sidebar': [{
      id: 'tech-specs',
      component: TechSpecsField as never,
      position: 80,
    }],
  },
})
```

If you don't see a slot at the right spot in the form, see [Slots catalog](/developer/dashboard/slots-catalog) — if one's missing, open a PR adding it.

## 2. Build the field component

The slot's context provides the React Hook Form instance for the page's form. Register your input against it:

```tsx
// packages/dashboard/src/fields/tech-specs-field.tsx
import {
  Card, CardContent, CardHeader, CardTitle,
  Field, FieldError, FieldLabel, Textarea,
} from '@spree/dashboard-ui'
import { useTranslation } from 'react-i18next'
import type { UseFormReturn } from 'react-hook-form'

interface Props {
  form: UseFormReturn<{ tech_specs: string }>
}

export function TechSpecsField({ form }: Props) {
  const { t } = useTranslation()
  const error = form.formState.errors.tech_specs?.message

  return (
    <Card>
      <CardHeader>
        <CardTitle>{t('admin.fields.product.tech_specs.label')}</CardTitle>
      </CardHeader>
      <CardContent>
        <Field>
          <FieldLabel className="sr-only" htmlFor="tech_specs">
            {t('admin.fields.product.tech_specs.label')}
          </FieldLabel>
          <Textarea
            id="tech_specs"
            rows={6}
            placeholder={t('admin.fields.product.tech_specs.placeholder')}
            aria-invalid={Boolean(error)}
            {...form.register('tech_specs', {
              maxLength: { value: 5000, message: 'Too long' },
            })}
          />
          {error && <FieldError>{error}</FieldError>}
        </Field>
      </CardContent>
    </Card>
  )
}
```

A few patterns worth flagging:

- **Use `form.register('tech_specs')`** — the parent form's `defaultValues` already includes `tech_specs` from the API response, so RHF wires up the initial value.
- **Use the field-key i18n convention** — `admin.fields.product.tech_specs.{label,placeholder}` falls back to `admin.fields.tech_specs.*` if you want to share the label across resources.
- **Apply `aria-invalid`** when the field has an error so screen readers announce it.

## 3. Extend the Zod schema (optional)

If your customization adds client-side validation, extend the product form schema in the host app:

```ts
// packages/dashboard/src/schemas/product.ts
import { z } from 'zod'

export const productFormSchema = z
  .object({
    name: z.string().min(1),
    // …existing core fields…
    tech_specs: z.string().max(5000).default(''),
  })
```

The form already uses `zodResolver(productFormSchema)`, so adding the key automatically runs validation. If the field has no client-side constraints, skip this step — server-side validation will round-trip through `mapSpreeErrorsToForm`.

## 4. Translations

```json
// packages/dashboard/src/locales/en.json
{
  "fields": {
    "product": {
      "tech_specs": {
        "label": "Technical specifications",
        "placeholder": "RAM, CPU, dimensions, weight…"
      }
    }
  }
}
```

i18next merges this into the existing `admin` namespace. See [Translations](/developer/dashboard/customization/translations) for the field-key fallback chain.

## What you should *not* do

- **Don't add a separate `useState` for the value.** The form's RHF instance owns it; a duplicate state goes out of sync on submit/reset.
- **Don't call `adminClient.products.update()` from inside the field.** The parent form's submit handler already does that — your job is to register the value.
- **Don't add a `useEffect` that copies `props.product.tech_specs` to local state.** RHF's `defaultValues` from the parent already handles the initial value.
