The dashboard's product form lives in `@spree/dashboard` and renders core attributes (name, description, price, …). When you've added a custom column to `spree_products` or a custom field through a [metafield definition](/developer/customization/metadata), you'll want to edit it on the same page. This recipe adds a **self-contained editor card** to the product page's sidebar: it owns its own small form and saves through the Admin API.

<Note>
Slot widgets are additive — the built-in product form doesn't expose its internal form instance to slots, so your card manages its own state and its own save. That's a feature: your field can't break the core form, and it works identically from a host app or a distributed plugin.
</Note>

## Backend prerequisites

This recipe assumes you've added the column or metafield on the Rails side:

```ruby
# A migration:
class AddTechSpecsToSpreeProducts < ActiveRecord::Migration[7.2]
  def change
    add_column :spree_products, :tech_specs, :text, null: false, default: ''
  end
end
```

And declared it as a permitted attribute on the Admin API serializer + controller (see [Backend integration](/developer/dashboard/customization/backend)).

## 1. Register the card

Inject it into the product form's sidebar via a slot:

```tsx
// src/plugins.ts (in your dashboard app)
import { defineDashboardPlugin } from '@spree/dashboard-core'
import { TechSpecsCard } from './fields/tech-specs-card'

defineDashboardPlugin({
  slots: {
    'product.form_sidebar': [{
      id: 'tech-specs',
      component: TechSpecsCard as never,
      position: 80,
    }],
  },
})
```

If you don't see a slot at the right spot in the form, see [Slots catalog](/developer/dashboard/slots-catalog) — if one's missing, open a PR adding it.

## 2. Build the card

The slot's context is `{ product }` — the product being edited. The card seeds its own React Hook Form instance from it and saves with its own button:

```tsx
// src/fields/tech-specs-card.tsx
import { adminClient, mapSpreeErrorsToForm, useResourceMutation } from '@spree/dashboard-core'
import {
  Button,
  Card, CardContent, CardFooter, CardHeader, CardTitle,
  Field, FieldError, FieldLabel, Textarea,
} from '@spree/dashboard-ui'
import { useForm } from 'react-hook-form'
import { useTranslation } from 'react-i18next'

interface Props {
  product: { id: string; tech_specs?: string }
}

interface FormValues {
  tech_specs: string
}

export function TechSpecsCard({ product }: Props) {
  const { t } = useTranslation()
  const form = useForm<FormValues>({
    defaultValues: { tech_specs: product.tech_specs ?? '' },
  })

  const update = useResourceMutation({
    mutationFn: (values: FormValues) => adminClient.products.update(product.id, values),
    invalidate: [['products']],
  })

  async function onSubmit(values: FormValues) {
    try {
      await update.mutateAsync(values)
    } catch (err) {
      if (!mapSpreeErrorsToForm(err, form.setError)) throw err
    }
  }

  const error = form.formState.errors.tech_specs?.message

  return (
    <Card>
      <CardHeader>
        <CardTitle>{t('admin.fields.product.tech_specs.label')}</CardTitle>
      </CardHeader>
      <form onSubmit={form.handleSubmit(onSubmit)}>
        <CardContent>
          <Field>
            <FieldLabel className="sr-only" htmlFor="tech_specs">
              {t('admin.fields.product.tech_specs.label')}
            </FieldLabel>
            <Textarea
              id="tech_specs"
              rows={6}
              placeholder={t('admin.fields.product.tech_specs.placeholder')}
              aria-invalid={Boolean(error)}
              {...form.register('tech_specs', {
                maxLength: { value: 5000, message: t('admin.fields.product.tech_specs.too_long') },
              })}
            />
            {error && <FieldError>{error}</FieldError>}
          </Field>
        </CardContent>
        <CardFooter>
          <Button type="submit" size="sm" disabled={update.isPending || !form.formState.isDirty}>
            {t('admin.common.save')}
          </Button>
        </CardFooter>
      </form>
    </Card>
  )
}
```

A few patterns worth flagging:

- **`invalidate: [['products']]`** refetches the product queries after a save, so the page (and list views) pick up the new value.
- **`mapSpreeErrorsToForm`** routes a 422 (e.g. a server-side length validation) onto the field as an inline error; `useResourceMutation` already suppresses its toast for validation failures.
- **Disable save until dirty** — `form.formState.isDirty` keeps the button inert when nothing changed.
- **Use the field-key i18n convention** — `admin.fields.product.tech_specs.{label,placeholder}`; see [Translations](/developer/dashboard/customization/translations#field-key-convention).
- **Apply `aria-invalid`** when the field has an error so screen readers announce it.

```ts
i18n.addResourceBundle('en', 'translation', en, true, true)
```

See [Translations](/developer/dashboard/customization/translations) for locales beyond English.

## What you should *not* do

- **Don't mirror the value into `useState` alongside RHF.** The card's form instance owns it; duplicate state goes out of sync after a save or reset.
- **Don't write to the core form's fields from your card.** The built-in form saves core attributes; your card saves yours. Two independent submits can't clobber each other.
- **Don't skip the server-side validation.** The client-side `maxLength` is UX; the Rails validation is the contract.
