The dashboard uses [i18next](https://www.i18next.com/) with the `react-i18next` bindings. The base bundle ships in `@spree/dashboard` under the `admin` namespace. You add your strings to the same `admin` namespace so they look and feel native — same `admin.fields.<resource>.<attribute>.{label,placeholder,help}` convention, same fallback chain.

## Add translations

```ts
import i18n from 'i18next'

i18n.addResourceBundle('en', 'admin', {
  reports: {
    title: 'Reports',
    empty: 'No reports yet.',
  },
  fields: {
    report: {
      name: { label: 'Report name', placeholder: 'e.g. Q4 sales by region' },
    },
  },
}, /* deep */ true, /* overwrite */ false)
```

`addResourceBundle` is additive when `deep: true, overwrite: false` — your keys merge into the existing `admin` namespace without clobbering the dashboard's defaults.

Repeat per locale:

```ts
import en from './locales/en.json'
import fr from './locales/fr.json'
import de from './locales/de.json'

for (const [locale, bundle] of Object.entries({ en, fr, de })) {
  i18n.addResourceBundle(locale, 'admin', bundle, true, false)
}
```

## Use them

```tsx
import { useTranslation } from 'react-i18next'

function ReportsHeader() {
  const { t } = useTranslation()
  return <h1>{t('admin.reports.title')}</h1>
}
```

In a registry entry, pass the key directly:

```ts
nav.add({
  key: 'reports',
  label: i18n.t('admin.reports.title'),
  path: '/reports',
})
```

Note: `i18n.t` resolves at the call site, so the label is a snapshot taken at registration time. Registry entries store plain strings — if the user switches language without a reload, registered labels keep their old language until re-registered. Components that call `useTranslation()` re-render on locale change; registry labels don't. If that matters for your entry, re-register it on the i18n `languageChanged` event.

## Field-key convention

Form labels, placeholders, and help text resolve via this fallback chain (defined in `@spree/dashboard-core`):

1. `admin.fields.<resource>.<attribute>.<facet>` — e.g., `admin.fields.report.name.label`
2. `admin.fields.<attribute>.<facet>` — e.g., `admin.fields.name.label`

The form helpers (`<Field>`, `<Input>`, `<FieldError>`) walk this chain automatically — you only pass the attribute name. Define keys under `fields.<resource>` for resource-specific labels and rely on `fields.<attribute>` shared defaults for everything else (`name`, `email`, `created_at`, …).

In dev mode, missing keys log to the console with the chain that was tried.

## Server-side error mapping

The dashboard's `mapSpreeErrorsToForm` helper translates 422 responses onto `form.formState.errors`. Backend `errors` come back as `{ "name": ["can't be blank"] }`, and the helper sets the field error using the **translated** version of the same key chain:

```ts
// Spree returns:
// { "errors": { "name": ["can't be blank"] } }

// The helper looks up these keys in order, taking the first one that resolves:
// 1. admin.errors.report.name.cant_be_blank
// 2. admin.errors.name.cant_be_blank
// 3. The raw server string "can't be blank"
```

Provide the localised string in `admin.errors.<resource>.<attribute>.<message_key>` if you want the user to see something other than the raw English message.

## Order of operations

Register translations **before** any code that calls `i18n.t()` at module-load time (registry labels, table titles, etc.). The framework's own bundles are loaded before your code runs, so the pattern is simply — top of the file:

```tsx
// src/plugins.ts (same pattern in a plugin package's entry module)
import { defineDashboardPlugin, i18n } from '@spree/dashboard-core'
import en from './locales/en.json'

i18n.addResourceBundle('en', 'admin', en, true, false)  // ① translations first

defineDashboardPlugin({ /* ② registrations may now use i18n.t(...) */ })
```
