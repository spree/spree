---
  Spree Store API & SDK Review — Competitive Analysis

  Executive Summary

  The Spree v3 API and SDK have a solid foundation — prefixed IDs, flat JSON, consistent snake_case, good TypeScript types, and a clean resource-oriented SDK. But there are meaningful
  gaps compared to Stripe (gold standard) and modern ecommerce competitors (Medusa v2, Saleor, Vendure).

  ---
  1. OpenAPI Spec Quality Issues

  Orphan schemas polluting the spec

  The Store API spec contains 19 orphan schemas that no endpoint references: StoreNewsletterSubscriber, StoreExport, StoreImport, StoreStockItem, StoreReport, StorePromotion, etc. These
  appear to be auto-generated by typelizer but serve no purpose in the Store API. Admin schemas (AdminCustomer, AdminOrder, AdminProduct, etc.) are also leaked into the Store spec,
  exposing sensitive field names like last_sign_in_ip, considered_risky, internal_note.

  Fix: Filter typelizer output to only include schemas referenced by Store API endpoints.

  PaginationMeta schema is incomplete

  The schema defines only 4 fields (page, limit, count, pages), but every response example includes 9 fields (from, to, in, previous, next are undocumented). SDKs and code generators
  will miss these fields.

  Missing "Markets" tag

  5 market endpoints use the Markets tag, but it's missing from the top-level tags array and x-tagGroups.

  Inconsistent order_token parameter naming

  Some endpoints use order_token, others use token for the same concept (guest order access). This is a real stumbling block for developers.

  Monetary amount type inconsistency

  - StoreShipment.cost → string ("100.0")
  - StoreShippingRate.cost → number (100.0)
  - StoreGiftCard.amount → number
  - StoreOrder.total → string

  Stripe consistently uses integer cents for all amounts. Medusa uses number. Pick one and be consistent.

  StorePayment.source_type enum has empty entry

  enum:
    - credit_card
    - store_credit
    - payment_source
    -                    # ← null/empty

  4. SDK-Specific Issues

  client.store.store.get() reads terribly

  The double .store.store. is confusing — first is the API scope, second is the resource. Consider aliasing to client.store.info.get() or client.store.storeInfo.get().

  SDK quickstart docs show non-existent methods

  The docs show client.store.orders.list() but this doesn't exist — it's client.store.customer.orders.list(). This will be developers' first frustration.

  publishableKey is required even for Admin-only usage

  The SpreeClientConfig interface marks publishableKey as required. Admin-only consumers must still provide one.

  No request/response interceptors

  No middleware pattern for:
  - Automatic token refresh on 401
  - Request/response logging
  - Analytics
  - Custom error transformation

  Stripe, Medusa, and most modern SDKs support interceptors.

  No auto-pagination

  Stripe:
  for await (const product of stripe.products.list()) { ... }
  Spree: manual page loop with meta.pages check. This is table-stakes DX in 2026.

  Inconsistent list response shapes

  Some lists return PaginatedResponse<T> (with meta), others return bare { data: T[] } (countries, currencies, locales, markets). Consumers can't rely on a uniform shape.

  StoreCountry.market typed as unknown

  No type safety for an important association.

  Admin client is an empty shell

  AdminClient is exported publicly but has zero functionality. This misleads developers who see it in autocomplete.

  ---
  5. Checkout Flow DX Issues

  No clear state machine documentation

  Vendure's AddingItems → ArrangingPayment → PaymentSettled makes the order lifecycle crystal clear. Spree's next/advance/complete endpoints don't communicate what states exist or what's
   required at each step. The API spec doesn't document the checkout state machine.

  No eligibleShippingMethods equivalent

  Vendure has a dedicated query for "what shipping methods can this order use?" Spree requires listing shipments, then looking at shipping rates — an indirect path.

  Payment Sessions abstraction is good but underdocumented

  The payment session flow (create → update → complete) is a solid abstraction that decouples payment providers, but the OpenAPI spec doesn't explain the flow or when each step is
  needed.

  7. Top Priority Recommendations

  Critical (breaks developer trust)

  1. Fix orphan/admin schemas in Store API spec — leaking AdminCustomer with last_sign_in_ip is a security smell
  2. Fix PaginationMeta schema — add the 5 missing fields
  3. Standardize order_token/token parameter naming — pick one
  4. Fix SDK docs — orders.list() doesn't exist, will be first thing developers try
  5. Standardize monetary types — all strings or all numbers, document the choice

  High Priority (competitive gap)

  6. Add request_id to every response — essential for debugging
  7. Add idempotency key support — at minimum for payment operations
  8. Add auto-pagination to SDK — for await iterator pattern
  9. Document the checkout state machine — what states exist, what transitions are valid, what's required at each step
  10. Add field selection — even basic fields=name,slug,price to reduce payload size (Medusa's +/-/* operators are the gold standard here)

  Medium Priority (DX polish)

  11. Add doc_url to error responses — link to relevant docs page
  12. Add rate limiting headers — X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
  13. Support nested expand — expand=line_items.variant.product
  14. Add sort parameter to all list endpoints — not just products
  15. Add request/response interceptors to SDK
  16. Rename client.store.store to client.store.storeInfo or similar
  17. Don't export empty AdminClient — or add a deprecation/coming-soon notice

  Nice to Have (forward-looking)

  18. API versioning strategy — date-based like Stripe, or at least a migration guide system
  19. Customer-facing returns/cancellation endpoints
  20. Full-text search endpoint — beyond Ransack _cont
  21. JWKS endpoint — for edge/CDN token verification
  22. Cursor-based pagination option — alongside offset for high-volume endpoints
